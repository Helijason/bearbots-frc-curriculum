Module 2 - Lesson 05

How do I debug a robot that isn't here?

Logging is what turns "it broke at competition" into "I can see exactly what happened, frame by frame, on my couch."
~45 minutes
Hardware: XRP robot
Prereq: Lesson 04 - The IO pattern
Java · AdvantageKit · AdvantageScope

In Lesson 04 you wrote three files that magically logged everything to AdvantageScope. You called Logger.processInputs() in periodic() and trusted it. Today, we open the hood and look at what's actually happening - because the moment something doesn't get logged, you'll need to know why.

Your brain is probably thinking this right now

"The robot drives. The graph shows numbers. Why are we still talking about logging?"

Because the bug that costs you the most points this season won't happen during practice. It'll happen during a quals match, in front of a referee, with no time to debug. Your only witness will be the log file. If you didn't log the right thing, or you logged it wrong, you'll never know what happened. Logging discipline is the difference between fixing a bug in 10 minutes and re-creating it for hours.

The 60-second recap of Lesson 04

The IO pattern split your subsystem into three files: an interface, an implementation, and the subsystem class. The point was hardware independence. The hidden bonus - the one we're going to use today - is that the IO pattern is also what makes deterministic logging and replay work. Same struct of inputs, every cycle, written to a file you can read later.

What @AutoLog actually does

You wrote @AutoLog at the top of DriveIOInputs in Lesson 04. You probably assumed it was decoration. It is doing real work.

@AutoLog is a Java annotation processor

When you compile your project, AdvantageKit's annotation processor sees @AutoLog on your class and generates a brand-new sibling class next to it called DriveIOInputsAutoLogged.

That generated class extends yours and adds two methods you never wrote: toLog(LogTable) and fromLog(LogTable). Those methods are how AdvantageKit serializes your inputs to the log file and reads them back during replay.

You never see the generated file in your editor. It lives in build/generated/. It's regenerated every time you build.

You write

@AutoLog
public class DriveIOInputs {
  public double leftPos;
  public double rightPos;
}

Compiler generates

public class DriveIOInputsAutoLogged extends DriveIOInputs {
  void toLog(LogTable t) { ... }
  void fromLog(LogTable t) { ... }
}

You use the generated one in your subsystem.

Why it matters

If @AutoLog is missing, no AutoLogged class exists. Your inputs aren't logged. AdvantageScope shows nothing. Your subsystem might still work - but you're flying blind.

Inputs vs outputs - two annotations, two purposes

Logging in AdvantageKit splits cleanly into two categories. Knowing which is which is half the battle.

Inputs - @AutoLog on a class

Things you read: encoder positions, gyro angles, current draw, button states.

Wrap them in a struct, slap @AutoLog on the struct, and AdvantageKit logs every field automatically.

These are the values that change because the world changed.

Outputs - @AutoLogOutput on a field

Things you decide: target velocity, calculated pose, autonomous mode name, "is the shooter ready?" booleans.

Put @AutoLogOutput on a field directly - AdvantageKit logs the value every time it changes.

These are the values that change because your code decided.

When to use which

Reading a sensor? It's an input. Goes in the Inputs struct.

Computing a value from sensor readings? It's an output. Goes on a @AutoLogOutput field, or use Logger.recordOutput("Drive/AveragePosition", value) directly.

Setting a motor voltage? The voltage you decided is an output (log it). The current the motor reports back is an input (log it via the inputs struct).

Inputs and outputs go to different folders in the AdvantageScope sidebar. You'll see RealOutputs/, ReplayOutputs/, and the AdvantageKit-prefixed input folders.

The order matters - updateInputs THEN processInputs

This is the mistake everyone makes once. Once.

The order is not optional

In your subsystem's periodic(), the two lines must be in this exact order:

io.updateInputs(inputs);                    // Step 1: read hardware into struct
Logger.processInputs("Drive", inputs);     // Step 2: log the struct (or read from log in REPLAY)

If you swap them, you log whatever was in the struct from the previous cycle - or zeros if it's the first cycle. Your data is always one tick stale. You will not notice this immediately. You will notice it three weeks later when your odometry is off by exactly 20 milliseconds and you can't figure out why.

Three modes, one code path

This is where AdvantageKit's architecture pays off. The same periodic() method behaves three different ways, controlled by Constants.currentMode:

REAL

updateInputs() reads from real hardware.

processInputs() writes the struct to a log file on the roboRIO USB drive.

What runs at competition.

SIM

updateInputs() reads from your physics sim.

processInputs() writes to a log file on your laptop.

What runs when you Ctrl+Shift+P → Simulate.

REPLAY

updateInputs() does nothing - the IO is a no-op stub.

processInputs() reads from the log file into the struct, overwriting whatever was there.

Magic.

Why replay is the killer feature

Imagine you saved the log from yesterday's match. The robot crashed at second 42. You're sitting at home with a laptop and no XRP. You set Constants.currentMode = REPLAY, hand AdvantageKit the log file, and run your robot code. Your subsystems get the exact same sensor inputs as during the match - byte for byte. You can put a breakpoint anywhere. You can change the logic and see what would have happened. You can do this on a plane.

This is impossible without disciplined logging. It's why we care about the order of those two lines.

Student
Why don't we just use SmartDashboard? It's simpler.
Fair question
SmartDashboard publishes to NetworkTables in real time - great for live dashboards, useless after the match. Once the robot powers off, the values are gone. AdvantageKit writes a binary log file that survives. You can replay it, scrub through it, share it with another programmer over Slack. Use both: SmartDashboard for live driver displays, AdvantageKit for everything you might want to debug later.
Student
Doesn't logging slow the robot down?
Negligibly
AdvantageKit logs at the same 50Hz rate as the robot loop. Each cycle is a few hundred numbers. The serialization is fast and the file write is buffered. On a real robot, log files end up around 2–5 MB per match. The performance hit is in the noise - you will never measure it.
Brain power - before you move on

Without looking back: name the two annotations you'd use, and explain when you'd reach for each one. Then say out loud: "updateInputs first, processInputs second." Memorize the order - it is the bug you will hit otherwise.

You've used AdvantageScope to drag a value onto a graph. That's about 5% of what it can do. The other 95% is the difference between "I can see the data" and "I can find any bug, any time, in any saved log." Today we cover the 95%.

Your brain is probably thinking this right now

"This is a lot of UI. Do I need to memorize all of it?"

No. You need to know which features exist, so when you have a problem, you know what to reach for. Reading the manual once is faster than re-discovering features under stress at competition.

Live mode vs log mode

AdvantageScope has two ways to get data, and switching between them is a one-click operation that students conflate constantly.

Live - Connect to Robot/Sim

Streams data as it happens. The timeline keeps growing. Use this when the robot or simulator is running right now.

File → Connect to Simulator (or to Robot)

Log - Open Log File

Loads a saved .wpilog file. The full timeline is available immediately - you can scrub anywhere, freely.

File → Open Log... (point at a file)

How to find your log files

Sim logs: by default in logs/ next to your project folder, named with the date and time.

Real-robot logs: on the USB stick plugged into the roboRIO. Pull the USB after the match, plug it into your laptop.

If you don't see a logs/ folder, the Logger isn't writing files - check Logger.addDataReceiver(new WPILOGWriter()) is in your Robot constructor.

The sidebar - how to find anything

The left sidebar in AdvantageScope is a tree of every key in the log. There's a lot. Knowing the structure speeds up everything.

RealOutputs/

Everything logged via Logger.recordOutput() when running on real hardware (or in SIM). This is where the values your code decided live: target velocities, computed poses, autonomous chooser selections.

ReplayOutputs/

The same outputs, recomputed during a replay run. If they don't match RealOutputs/ for the same timestamp, your code's behavior changed - on purpose (you fixed a bug) or by accident (you broke something).

AdvantageKit/RealOutputs/ + per-subsystem folders

Inputs from your @AutoLog structs - e.g., Drive/LeftPositionMeters, Shooter/CurrentAmps. The folder name comes from the first arg of Logger.processInputs("Drive", inputs).

NT/ and DS/

Anything published to NetworkTables (SmartDashboard, Shuffleboard) and driver-station state. Useful for cross-checking what the dashboard saw vs. what the robot actually did.

The visualization tabs

The big buttons across the top of AdvantageScope each open a different way of looking at the same data. The ones you'll use most:

Line Graph

Y-axis: numeric value. X-axis: time. Drag any number-valued key onto it. The 90% case.

Multiple keys on one graph let you compare directly.

Odometry / 2D Field

Drop a Pose2d or array of poses onto a top-down field view. See where the robot was and where it thought it was.

Best way to debug autonomous routines.

Table

Scrubbable spreadsheet view of every value at every timestamp. Slow for long logs but unbeatable for "what was the value at exactly t=42.0?"

Console

All System.out.println() output from the robot. If you printed a debug message during a match, this is where it shows up.

Time scrubbing

How to scrub through a log

1. Open a log file (or pause a live connection)

2. The timeline at the bottom of the window shows the full duration

3. Click anywhere on the timeline - every visualization jumps to that moment

4. Use arrow keys to step frame-by-frame (one robot loop per arrow press)

5. The play button replays at real-time speed; speed slider goes faster or slower

This is how you find the exact moment something went wrong.

Replay mode - running new code on old data

Replay is the killer feature. It lets you change your code, re-run it against a saved log, and see what would have happened - without touching the robot.

How to run a replay

1. In Constants.java, set simMode = Mode.REPLAY

2. Set the path to the log file you want to replay (the project README has the exact line)

3. Run WPILib: Simulate Robot Code as normal

4. Your subsystems get the exact same input values as during the original run

5. A new log gets written, with ReplayOutputs/ populated from your code

6. Open the new log in AdvantageScope - RealOutputs/ and ReplayOutputs/ let you compare

Replay only works if logging was disciplined

If a sensor reading was never logged, replay can't reproduce it. If processInputs() wasn't called, the log is empty. If @AutoLog was missing, the struct didn't get serialized. Bad logging during the match means no debugging after - this is why the rules in Part 1 are not optional.

Silver - add a new logged input

Add a derived value to your inputs class so it gets logged automatically. The XRP doesn't have a current sensor - instead, log a calculated total distance.

  • Open DriveIOInputs.java
  • Add a new field: public double totalDistanceMeters = 0.0;
  • In DriveIOXRP.java's updateInputs() method, set the field to (left + right) / 2.0
  • Build the project, simulate, drive for 10 seconds
  • Open the new log - verify Drive/TotalDistanceMeters appears in AdvantageScope
If you added the field but it doesn't appear in AdvantageScope, did you rebuild the project? @AutoLog is processed at compile time. Editing the inputs class without rebuilding means the AutoLogged class is stale. 
// In DriveIOInputs.java
@AutoLog
public class DriveIOInputs {
    public double leftPositionMeters = 0.0;
    public double rightPositionMeters = 0.0;
    public double totalDistanceMeters = 0.0;   // new field
}

// In DriveIOXRP.java's updateInputs():
inputs.leftPositionMeters = drivetrain.getLeftDistanceInch() * 0.0254;
inputs.rightPositionMeters = drivetrain.getRightDistanceInch() * 0.0254;
inputs.totalDistanceMeters =
    (inputs.leftPositionMeters + inputs.rightPositionMeters) / 2.0;
Gold - run a replay

Take a log file from a sim run, switch to replay mode, and re-run your code against it. Verify the outputs match.

  • Drive in sim for ~10 seconds. Save the resulting log file.
  • In Constants.java, change simMode from Mode.SIM to Mode.REPLAY
  • Set the replay log path to point at the file you just saved (check the project README for the exact constant name)
  • Run WPILib: Simulate Robot Code
  • Open the resulting new log in AdvantageScope
  • Find a value that exists in both RealOutputs/ and ReplayOutputs/ - do they match?
  • Bonus: change one line of code in your subsystem (e.g., scale a value by 2). Re-run the replay. Verify the ReplayOutputs/ value now differs from RealOutputs/ in exactly the way you expected.
Replay outputs match real outputs when your code is unchanged - that's how you know replay is working. The interesting case is when they don't match: that tells you the change you made affected behavior. If they always match no matter what you do, your replay isn't actually running - double-check the mode constant and log path.

Bug 1 - missing @AutoLog annotation

The class looks innocent. It even has fields. It's just not being processed by anyone.

Without @AutoLog at the top of DriveIOInputs, the AdvantageKit annotation processor never runs on it. The generated DriveIOInputsAutoLogged class doesn't exist. Logger.processInputs() needs that generated class to know how to serialize the struct. Without it, your inputs are silently never logged. AdvantageScope's Drive folder is empty.

Bug 2 - processInputs called BEFORE updateInputs

The lines are in the wrong order. The robot logs whatever was in the struct from last cycle. Or zeros. Forever.

On the first call to periodic(), the inputs struct is full of default zeros. processInputs("Drive", inputs) writes those zeros to the log. updateInputs(inputs) then reads real values into the struct - but it's too late, the log has already been written for this cycle. Next cycle, you log last cycle's values. Your data is always one tick behind. The fix: swap the lines.

Fix
io.updateInputs(inputs);
Logger.processInputs("Drive", inputs);

Bug 3 - output key missing the subsystem prefix

The value gets logged, but it goes to the root namespace where you'll never find it - unless you know to look for the lonely "avgPosition" floating around with no folder.

The first argument to Logger.recordOutput() is the key path in the log. Conventions matter: every output should be prefixed with its subsystem name. "avgPosition" dumps the value into the root, where it will collide with any other subsystem that names something the same. "Drive/AvgPosition" puts it neatly inside the Drive folder where it belongs. AdvantageScope's sidebar tree is built on these slashes.

Fix
Logger.recordOutput("Drive/AvgPosition", avg);
All 3 bugs found. Missing annotation, swapped order, missing prefix - these are the three logging bugs you'll see most often in real team code. The first two break logging entirely. The third buries your data where you can't find it. Fix them on sight from now on.
