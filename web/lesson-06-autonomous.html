Java · WPILib commands

Autonomous mode is the first 15 seconds of every match. The driver can't touch the controller. The robot has to know what to do, when to stop doing it, and when to do something else. None of that is possible if your code is just a list of motor commands. You need a structure that has a beginning, a middle, and an end - and you need to compose those structures into longer sequences.

That structure is a Command. Today we learn what one looks like, why it's better than the alternatives, and how to combine them into routines that actually finish.

Your brain is probably thinking this right now

"Why don't I just put the autonomous code in autonomousPeriodic()? It runs every 20ms. I'll set the motors, then check the encoder, then turn off the motors when the distance is reached. Done."

You can do that. It works for one move. The second you want to chain moves - drive, then turn, then drive again - you're writing a state machine by hand. Every team that has tried this has eventually rewritten it as commands. Save yourself the rewrite.

The four questions every command answers

A command is a Java class that extends Command and answers four questions about what the robot should do. That's it. Once you understand these four questions, you understand commands.

1. initialize()

"What do I do at the very start?"

Runs once. Use it to record a starting position, reset a sensor, set a target value - anything that needs to happen before the loop begins.

Forgetting to record a start position is bug #1.

2. execute()

"What do I do every cycle?"

Runs every 20ms while the command is scheduled. Set motor outputs here. This is the body of the loop.

Anything heavy here runs 50 times per second.

3. isFinished()

"Am I done yet?"

Runs every cycle, after execute(). Return true when the command should stop. Until you return true, the command keeps going.

Returning false always = command never ends.

4. end(boolean interrupted)

"What do I clean up?"

Runs once when the command stops - whether isFinished() returned true or another command interrupted it. Stop the motors here.

Forgetting to stop motors is bug #2.

What that looks like in code

public class DriveDistance extends Command {
    private final DriveSubsystem drive;
    private final double targetMeters;
    private double startMeters;     // where we began

    public DriveDistance(DriveSubsystem drive, double meters) {
        this.drive = drive;
        this.targetMeters = meters;
        addRequirements(drive);                  // "I need exclusive access to drive"
    }

    @Override public void initialize() {
        startMeters = drive.getLeftPositionMeters();      // remember the start
    }

    @Override public void execute() {
        drive.drive(0.5, 0.0);                            // keep driving forward
    }

    @Override public boolean isFinished() {
        double traveled = drive.getLeftPositionMeters() - startMeters;
        return traveled >= targetMeters;                // done?
    }

    @Override public void end(boolean interrupted) {
        drive.stop();                                     // always stop motors
    }
}

Read it again, slowly. Match each method to one of the four questions. addRequirements(drive) is the only odd one - that's the command telling WPILib which subsystem it needs exclusive access to (more on that below).

Why this beats autonomousPeriodic()

Composable

Once DriveDistance works, you can build SequentialCommandGroup(driveDist, turn, driveDist) in one line. Try doing that with state-machine code.

⏱ Interruptible

Driver hits a button to abort? The scheduler ends the command, calls end(true), motors stop. With autonomousPeriodic(), you'd build that yourself. Probably wrong.

Reusable

The same DriveDistance works in autonomous, in teleop on a button bind, in a test routine. Write it once. Use it everywhere.

Loggable

The CommandScheduler logs every initialize, execute, and end. Open AdvantageScope, look at the Commands folder - you can see exactly which command was running, when. Compare to periodic(): nothing.

The CommandScheduler - what calls these methods?

You don't call initialize() yourself. WPILib has a CommandScheduler that runs every 20ms (the same line you put in robotPeriodic() back in Lesson 1). When you "schedule" a command - usually with command.schedule() or by binding it to a button - the scheduler takes ownership of running it.

The scheduler's lifecycle, every loop

1. For each scheduled command, if it just got scheduled this cycle: call initialize().

2. Call execute().

3. Call isFinished(). If it returns true → call end(false) and remove from schedule.

4. If a higher-priority command takes the same subsystem → call end(true) on the loser, schedule the winner.

That's the whole loop. Every command, every cycle. You write the four methods; the scheduler does the rest.

Subsystem requirements - why addRequirements() matters

addRequirements(drive) tells the scheduler "this command needs the drivetrain - nothing else can use it while I'm running." If a second command tries to grab the same subsystem, the scheduler interrupts the first one (calls end(true)) and starts the new one.

Forget addRequirements and bad things happen

If DriveDistance doesn't declare a requirement on drive, and another command also drives the motors, the scheduler runs both. Both call drive.drive() in execute(). Whichever ran last wins for that cycle - effectively, your motors get random commands. The robot does something neither command intended.

Always add requirements. Always.

Student
Why is end() called even if I get interrupted? I didn't finish.
Reason
Because cleanup matters either way. If you were driving forward and got interrupted, the motors are still running - end() needs to stop them or the robot keeps going. The interrupted boolean is just a hint about why you're ending. The cleanup is the same.
Student
What if my command really is instant - like "reset gyro"?
Use InstantCommand
WPILib provides InstantCommand for one-shot actions. new InstantCommand(() -> gyro.reset(), drive) does what you'd expect: runs the lambda once, requires drive, ends immediately. No need to write a class.

Out loud, name the four methods of a command and what each does. Then explain why end() exists separately from isFinished() - couldn't they be combined? (Answer: no - end() also runs when interrupted. isFinished() only signals "I'm ready to be done.")

One command runs one move. Auto routines need multiple moves - drive forward, turn, drive forward, score. The way you combine commands is called composition, and it's the actual point of the command framework. If you understand composition, you can build any auto.

Your brain is probably thinking this right now

"Can't I just chain commands by scheduling the next one in end()?"

You can. People have. It works for two commands and falls apart at three. Composition is what WPILib gives you instead. It's what every team is using by championships. Learn it now.

SequentialCommandGroup - one after another

This is the workhorse. Pass it commands in order; the group runs them one at a time, only moving to the next when the current one's isFinished() returns true.

public Command driveLShape(DriveSubsystem drive) {
    return new SequentialCommandGroup(
        new DriveDistance(drive, 1.0),       // drive 1m forward
        new TurnToAngle(drive, 90.0),        // then turn 90°
        new DriveDistance(drive, 1.0)        // then drive another 1m
    );
}

That's a complete autonomous routine. Three commands, three lines, a robot that draws an L on the carpet. The group itself is also a command - it has its own isFinished() that returns true when the last child finishes. So you can put a SequentialCommandGroup inside another SequentialCommandGroup if you want.

ParallelCommandGroup - all at once

Sometimes you want two commands running at the same time - drive forward while spinning up the shooter. ParallelCommandGroup runs them concurrently and finishes when all children finish.

return new ParallelCommandGroup(
    new DriveDistance(drive, 2.0),
    new SpinUpShooter(shooter, 4000)
);
Parallel groups can't share subsystems

If both children require the same subsystem, the scheduler will refuse to run them in parallel - only one command can own a subsystem at a time. Two commands needing the drivetrain in parallel? Doesn't work. Group them sequentially or compose them differently.

Two cousins of ParallelCommandGroup

ParallelDeadlineGroup

"Run all of these together, but the first command (the deadline) decides when we're done. When it finishes, kill the rest."

Use case: drive for 3 seconds while running the intake. When the drive finishes, stop the intake too.

ParallelRaceGroup

"Run all of these together. As soon as any of them finishes, kill the rest."

Use case: try to score, but cancel after 2 seconds (a WaitCommand(2) running alongside the score command).

Decorators - the fluent shortcuts

Building command groups gets verbose. WPILib provides decorator methods on the Command class that read like English and produce the same composed structures.

// Same as new SequentialCommandGroup(a, b):
a.andThen(b);

// Same as new ParallelCommandGroup(a, b):
a.alongWith(b);

// Run a until 3 seconds pass, whichever comes first:
a.withTimeout(3.0);

// Run a until some condition becomes true:
a.until(() -> someBoolean);

// Chain them — these all return Commands you can keep decorating:
driveDist.andThen(turn).andThen(driveDist).withTimeout(15.0);

The AutoChooser pattern

One robot, many possible auto routines, one match to pick. The AutoChooser is a SmartDashboard widget that lets a driver pick which routine to run before the match starts. The pattern is short and worth memorizing.

// In RobotContainer.java
private final SendableChooser<Command> autoChooser = new SendableChooser<>();

public RobotContainer() {
    autoChooser.setDefaultOption("Drive 1m", new DriveDistance(drive, 1.0));
    autoChooser.addOption("L-shape", driveLShape(drive));
    autoChooser.addOption("Score and back up", scoreAndBackUp(drive, shooter));
    SmartDashboard.putData("Auto Choice", autoChooser);
}

public Command getAutonomousCommand() {
    return autoChooser.getSelected();
}

Then in Robot.java's autonomousInit(): m_autonomousCommand = robotContainer.getAutonomousCommand(); m_autonomousCommand.schedule();. The chooser appears as a dropdown in SmartDashboard or AdvantageScope; whatever the driver picks is what runs.

A good autonomous lifecycle

autonomousInit() → pull command from chooser → schedule it.

autonomousPeriodic() → usually empty - the scheduler is already running the command.

teleopInit() → if the auto command is still running, cancel it (otherwise it'll fight the driver).

Most of this is already in the starter project. You just write the commands.

Pattern recognition

Once you've internalized commands, almost every autonomous looks like this: a SequentialCommandGroup at the top, with each child being either a primitive command (DriveDistance, TurnToAngle) or a smaller group. Maybe a parallel group somewhere to run two mechanisms at once. Maybe a WaitCommand for delays. That's it. Once you've seen this shape three times, you'll see it everywhere - including in real team code at competition.

Bronze - run the existing DriveDistance command

The starter project already has DriveDistance and a working AutoChooser. Verify the chain works end-to-end and observe what the scheduler logs.

  • Run the simulator. Open AdvantageScope and connect.
  • Set Robot State to Autonomous → Enable. Watch what the robot does.
  • In AdvantageScope's sidebar, expand the Commands or SmartDashboard/Auto Choice folder. Find the timestamp where the command initialized and where it ended.
  • Find Drive/LeftPositionMeters in the log. What position did the robot reach when the command ended? Does it match the target distance?

If you can't find the Commands folder in AdvantageScope, the scheduler logging may not be enabled - look for a CommandScheduler.getInstance().onCommandInitialize(...) setup in Robot.java.
Silver - write a TurnToAngle command

Write a new command that turns the robot by a given number of degrees relative to its starting heading, then ends. Use the same shape as DriveDistance.

  • Create TurnToAngle.java in the commands folder. Extend Command.
  • Constructor takes DriveSubsystem drive and double targetDegrees. Call addRequirements(drive).
  • initialize(): record the starting gyro angle
  • execute(): drive with zero forward, some rotational input (try 0.5)
  • isFinished(): return true when the absolute change in heading reaches the target
  • end(): stop the drivetrain
  • Add new TurnToAngle(drive, 90.0) as an option in the AutoChooser. Run it. Verify in AdvantageScope.
Two gotchas. First: the gyro might wrap around 180° or 360°. For small angles (under 180°) you can ignore wrap-around, but check that you handle a 90° turn correctly. Second: rotational sign matters. Positive rotation might mean clockwise on your robot or counter-clockwise - don't assume. Test both and pick the convention that matches the gyro reading. 
public class TurnToAngle extends Command {
    private final DriveSubsystem drive;
    private final double targetDegrees;
    private double startDegrees;

    public TurnToAngle(DriveSubsystem drive, double degrees) {
        this.drive = drive;
        this.targetDegrees = degrees;
        addRequirements(drive);
    }

    @Override public void initialize() {
        startDegrees = drive.getHeadingDegrees();
    }

    @Override public void execute() {
        double rotation = Math.signum(targetDegrees) * 0.5;
        drive.drive(0.0, rotation);
    }

    @Override public boolean isFinished() {
        double turned = drive.getHeadingDegrees() - startDegrees;
        return Math.abs(turned) >= Math.abs(targetDegrees);
    }

    @Override public void end(boolean interrupted) {
        drive.stop();
    }
}
Gold - compose an L-shape route

Build a routine that drives forward 1m, turns 90°, then drives forward another 1m. Use a SequentialCommandGroup (or the .andThen() decorator). Add it to the AutoChooser. Verify the log shows three commands running back-to-back.

  • In RobotContainer.java (or a new AutoRoutines.java), write a method driveLShape(DriveSubsystem drive) that returns a Command
  • The returned command runs three children in sequence: drive 1m, turn 90°, drive 1m
  • Add it to the AutoChooser as "L-shape"
  • Run it in sim. Confirm in AdvantageScope: three command init events, three end events, robot ends at roughly (1, 1) relative to start
  • Bonus: add a new WaitCommand(0.5) between each move. Verify the gaps appear in the log.
  • Double bonus: wrap the whole thing in .withTimeout(15.0). Do you see the timeout limiting the route if you make one of the moves impossibly long?
If the L-shape ends prematurely, one of your child commands is finishing too early - probably TurnToAngle with a sign issue. If it never ends, one is finishing too late - probably DriveDistance with a wrong target. The Commands folder in AdvantageScope shows you exactly which child is hanging.

Bug 1 - ParallelCommandGroup instead of SequentialCommandGroup

All three commands fire at once. They all want the drivetrain. The scheduler can only give it to one. The other two get rejected. The robot ends up doing whichever one happened to win the race.

Two of the three children require the same subsystem (the drivetrain). A ParallelCommandGroup tries to run them all simultaneously, but the scheduler only allows one command per subsystem at a time - so the group fails to start, or only the first child runs. Either way: not what was intended. The fix is one word.

Fix
return new SequentialCommandGroup(...)

Bug 2 - isFinished() always returns false

The command runs forever. The robot drives off the field. Then off the table. Then off the planet. Theoretically.

The command needs to return true when the robot has traveled the requested distance. Hard-coded false means it never reaches that condition - the command stays scheduled, execute() keeps running, motors keep going. The next command in the sequence never starts.

Fix
return drive.getLeftPositionMeters() - startMeters >= targetMeters;

Bug 3 - end() doesn't stop the motors

The command technically ends. The scheduler removes it. But the last value sent to the motors is still 0.5. The motors keep running. The robot keeps driving. The command's gone but its motor commands aren't.

When a command ends - either by finishing or being interrupted - end() is your one chance to clean up. Motors don't auto-stop just because the command stopped calling execute(). Whatever they were last set to, they stay there. Always call drive.stop() (or set the motors to 0) in end().

Fix
@Override public void end(boolean interrupted) { drive.stop(); }
All 3 bugs found. Wrong group type, missing termination condition, missing cleanup - these are the three bugs you'll hit when writing your first auto. Once you've seen them, you'll spot them in seconds. The next time someone on your team writes a command that "doesn't end", you'll know exactly where to look.
