Module 2 - Lesson 02

What is a subsystem?

Or: why your robot can't just be one giant pile of code held together by hope and zip ties.
~45 minutes
Hardware: none — VSCode + simulator
Tools: VSCode WPILib tools, Simulator, AdvantageScope

It's week 5. Your robot has a drivetrain, an indexer, a shooter, and a climber. Every mechanism is controlled by code. If you put all that code in one file, that file is about 1,200 lines long. Two programmers are trying to edit it at the same time. Nobody is sure which lines control what. The robot match starts in 10 minutes.

This is the situation that subsystems exist to prevent.

Your brain is probably thinking this right now

"That sounds like a people problem, not a code problem. Can't everyone just be careful?"

They can try. They will fail. Code structure isn't about discipline — it's about making the right thing easy and the wrong thing hard. Subsystems make it structurally impossible to accidentally edit the shooter while you're debugging the climber.

The cafeteria analogy

Imagine your school cafeteria. There's a pizza station, a salad bar, and a dessert section. Each one does its own job. The pizza station doesn't also handle dessert. If the pizza oven breaks, the ice cream is fine.

Your robot works the same way. Each subsystem owns exactly one mechanism and controls it completely. If the shooter breaks, the drivetrain keeps driving.

Drivetrain

Moves the robot around the field

Gets blamed for everything

Indexer

Moves game pieces through the robot

Jams exactly when you need it not to

Shooter

Launches game pieces at a target

Always "almost working" during comp

Climber

Lifts the robot at end of match

Never tested until week 6

Three reasons that aren't just "because mentor said so"

Reason 1 — One job, one place

If the indexer is broken, you look in one file. Not everywhere. When you're debugging at 2am before competition, "one file" feels like a gift from the universe.

Reason 2 — Teammates can work in parallel

Two people editing the same file is a merge conflict waiting to ruin someone's day. With subsystems, one programmer works on the shooter while another works on the drivetrain — different files, no conflict.

Reason 3 — WPILib enforces it

The CommandScheduler prevents two commands from using the same subsystem at the same time. If one command is driving, it won't let another command accidentally also try to drive. This prevents the robot from doing contradictory things simultaneously. It has better conflict resolution than most group projects.

What a subsystem looks like in code

public class Indexer extends SubsystemBase {

    // All indexer code lives here.
    // Nothing else touches the indexer motor.
    // If someone adds drivetrain code here,
    // that person buys the team snacks.

    @Override
    public void periodic() {
        // Runs every 20ms. Update sensors, log data.
    }
}
Extending SubsystemBase registers this subsystem with the CommandScheduler automatically. That's the only requirement.

LoggedRobot, TimedRobot, and the command framework

A question you might have from Lesson 01: if we extend LoggedRobot, where does the command framework come from?

Student
Is there a CommandRobot class we're supposed to extend?
Good question, common confusion
No. The inheritance chain is: your Robot extends LoggedRobot, which extends TimedRobot. The command framework isn't a base class — it's a system that runs because you call CommandScheduler.getInstance().run() in robotPeriodic(). That one line is what makes the whole framework go.
Student
So the command framework isn't built into the robot class at all?
Exactly
It's opt-in. You could have a TimedRobot with no commands at all — just raw periodic methods. Command-based is a pattern we choose to use by running the scheduler. AdvantageKit's LoggedRobot wraps each loop cycle to capture data. These are two separate systems that work together.
There are no dumb questions

Q: Can a subsystem control two mechanisms?

Technically yes. It's almost always a bad idea. If two mechanisms are so tightly coupled that they must always be controlled together, they might belong in one subsystem. Otherwise separate them.

Q: How does a subsystem communicate with another subsystem?

Through commands in RobotContainer. Subsystems don't talk to each other directly — that would create dependencies that make debugging much harder. Commands coordinate between subsystems.

Q: What's the difference between periodic() and a command?

periodic() runs every loop cycle regardless — use it for reading sensors and logging. Commands run when scheduled — use them for actions triggered by buttons or autonomous. Sensor reading goes in periodic(). Motor control goes in commands.

Brain power — before you move on

Your robot has a turret that rotates horizontally and an arm that raises vertically. They're on the same mechanism. Are they one subsystem or two? Write your answer and your reasoning before reading Part 2. There's no single right answer — but there is a better answer.

Always use the WPILib tools to create subsystem files. Never create them by hand. The tools generate the right package declaration, imports, and class structure — getting these wrong causes confusing build errors that waste time.

Creating a subsystem file

How to create a new subsystem using WPILib tools

1. In VSCode's Explorer panel, right-click the folder where you want the file

    (e.g., src/main/java/frc/robot/subsystems/)

2. Select Create a new class/command

3. Select SubsystemBase from the list

4. Type the name of your subsystem (e.g., Indexer)

5. Press Enter — the file is created with the correct structure

Alternatively: Ctrl+Shift+P → WPILib: Create a new class

What the tool generates

The generated file looks exactly like this. Understand every line before adding anything:

package frc.robot.subsystems;  // matches your folder structure

import edu.wpi.first.wpilibj2.command.SubsystemBase;

public class Indexer extends SubsystemBase {

    public Indexer() {
        // Constructor — runs once when subsystem is created
        // Good place to configure hardware (motors, sensors)
    }

    @Override
    public void periodic() {
        // Runs every 20ms (50Hz)
        // Read sensors, log data, update state
        // DO NOT put motor control here — use commands
    }
}
Common mistake — motor control in periodic()

New students often put motor control code directly in periodic(). This runs every loop cycle regardless — the motor will run forever with no way to stop it. Motor control belongs in commands. Sensor reading and logging belong in periodic().

Registering your subsystem in RobotContainer

Creating the file isn't enough. You need to instantiate it in RobotContainer — otherwise it never exists.

public class RobotContainer {

    // Declare subsystems here
    private final Indexer indexer = new Indexer();

    public RobotContainer() {
        configureBindings();
    }

    private void configureBindings() {
        // Wire buttons to commands here
        // controller.a().onTrue(indexer.someCommand());
    }
}

Verifying your subsystem in the simulator

After creating and registering a subsystem, verify it loaded correctly before writing any logic:

Subsystem verification workflow

1. Build: Ctrl+Shift+P → WPILib: Build Robot Code — must succeed with no errors

2. Simulate: Ctrl+Shift+P → WPILib: Simulate Robot Code → select halsim_gui

3. Connect AdvantageScope: File → Connect to Simulator

4. Enable in Teleop in the Simulation GUI

5. In AdvantageScope's left panel, look for your subsystem's log key folder

If the folder appears - your subsystem is loaded and logging correctly.

If the folder doesn't appear - check that you instantiated it in RobotContainer AND that Logger.processInputs() is called in periodic().

Driving in simulation to verify commands work

Full sim + AdvantageScope workflow

1. Launch simulator → Simulation GUI opens

2. Joysticks panel: drag Keyboard 0 (or your controller) to Joystick[0]

3. Open AdvantageScope → File → Connect to Simulator

4. Robot State panel → Teleoperated → Enable

5. Click back on the Simulation GUI window (keyboard only works when this window is focused)

6. Drive with W/A/S/D keys or controller left stick

7. In AdvantageScope, drag Drive/LeftPositionMeters to a graph - should move when driving

Do this right now

Create a new subsystem called Indexer using the WPILib tool. Register it in RobotContainer. Build, simulate, connect AdvantageScope, and verify the build succeeds without errors. Don't add any logic — just confirm the subsystem exists and loads.

Bronze - identify and create

Given the description below, identify which mechanisms should be subsystems. Then create the file for one of them using the WPILib tool.

Robot description

Your robot has: two drive motors, a motor that spins a flywheel to shoot balls, a servo that adjusts the shooter angle, a sensor that detects when a ball is loaded, and an LED strip that shows robot status.

  • List every mechanism. Which ones should be separate subsystems? Which ones might share a subsystem?
  • Create the Shooter subsystem file using WPILib: Create a new class → SubsystemBase
  • Register it in RobotContainer. Build and verify no errors.
  • Add one System.out.println() inside periodic(). Run the sim. Does it print?
Think about which mechanisms need to be controlled independently. The servo adjusts the shooter angle — does it make sense to control it separately from the flywheel, or are they always used together?
Suggested: DriveSubsystem (two drive motors), ShooterSubsystem (flywheel + angle servo — they're tightly coupled), LEDSubsystem (optional — LEDs often get their own subsystem for state management). The ball sensor belongs inside whichever subsystem needs to read it, not its own subsystem.
Silver - build a minimal working subsystem

Create a complete minimal subsystem that: stores a goal state, changes state via a command, and logs the current state to AdvantageScope.

  • Create IndexerSubsystem using the WPILib tool
  • Add an enum IndexerGoal with values IDLE and ACTIVE
  • Add a private IndexerGoal goal field defaulting to IDLE
  • Add @AutoLogOutput annotation to the goal field so it appears in AdvantageScope
  • Add a setGoalCommand(IndexerGoal goal) method returning a runOnce() command
  • Wire a controller button to setGoalCommand(IndexerGoal.ACTIVE) in RobotContainer
  • Run sim, drive, press the button — watch the goal change in AdvantageScope
@AutoLogOutput is an AdvantageKit annotation — add it directly above the field declaration. For the controller binding in RobotContainer: driverController.a().onTrue(indexer.setGoalCommand(IndexerGoal.ACTIVE));. The goal should appear in AdvantageScope as IndexerSubsystem/Goal.
Gold - design the full subsystem structure

Given this season's game, design the complete subsystem architecture before writing any code. This is what senior programmers do before week 1.

  • List every mechanism on the robot (real or hypothetical)
  • Group them into subsystems with clear ownership boundaries
  • For each subsystem, define: what it reads (sensors/inputs), what it controls (motors/outputs), what goals/states it has, and what commands it exposes
  • Identify any cases where two subsystems need to coordinate — how do they do it without talking directly?
  • Create the file structure using WPILib tools — all subsystem shells, no implementation yet
  • Verify the project builds with all shells registered in RobotContainer
Subsystems coordinate through commands, not direct method calls. If the shooter needs to wait for the indexer to be ready, that's a command that requires both subsystems — not a method call from one subsystem to another. That distinction matters for scheduling.

Someone put all the robot logic in one place. The robot compiles. It even runs. But something is deeply wrong with the structure — and that wrongness will cause real problems the moment the robot gets more complex. Find the 3 structural bugs hiding in the highlighted lines.

Bug 1 - Indexer motor living inside DriveSubsystem

It was convenient. It is not correct. "Convenient" and "correct" are different things that feel the same until 11pm before competition.

The indexer motor belongs in an IndexerSubsystem — not in DriveSubsystem. When the indexer breaks, you'll search DriveSubsystem for the bug. When a command tries to run the indexer, it will accidentally require the drive subsystem, preventing the robot from driving at the same time. Move the indexer motor to its own subsystem.

Fix
Create IndexerSubsystem.java (WPILib tool → SubsystemBase) and move indexerMotor there.

Bug 2 - Motor control in periodic()

The robot drives forward. Forever. Without stopping. Even in autonomous. Even when disabled. This is what "always" means in robot code.

Motor control in periodic() runs every loop cycle regardless of mode or state. The robot will drive forward the moment it's enabled and never stop. Motor control belongs in commands — which can be started, stopped, and interrupted. periodic() is for reading sensors and logging, not commanding actuators.

Fix
Move motor control into a command. Set it as the default command in RobotContainer.

Bug 3 - runIndexer() requires DriveSubsystem, not IndexerSubsystem

The comment says "requires DriveSubsystem — correct!" The comment is not correct.

Because runIndexer() is defined inside DriveSubsystem, WPILib treats it as requiring the drive subsystem. This means running the indexer will lock out any drive commands — the robot can't drive while indexing. Commands should live in the subsystem that owns the hardware they control. Indexer commands belong in IndexerSubsystem.

Fix
Move runIndexer() to IndexerSubsystem. It will then correctly require only the indexer.
All 3 bugs found. Wrong subsystem ownership, motor control in periodic(), and misattributed command requirements — these are the three structural mistakes that haunt teams for entire seasons. You'll recognize all three on sight. That's genuinely useful.
