Module 2 - Lesson 04

Why does the robot need two files for one motor?

Or: the question every student asks, followed by the answer that actually makes sense.
~45 minutes
Hardware: XRP robot
Prereq: Lesson 02 — Subsystems
Java · AdvantageKit

You're staring at the project. There's DriveSubsystem.java, DriveIO.java, and DriveIOXRP.java. They all seem to be about driving. You're pretty sure you only have one drivetrain.

Your brain is correct. You do only have one drivetrain. So why three files?

Here's the short answer: your subsystem should not know what hardware it's running on.

Your brain is probably thinking this right now

"That sounds insane. Of course the drivetrain code knows it's a drivetrain. I can see the motors right there."

Totally fair. Keep reading. We'll get there.

Imagine this scenario

It's 11pm. Competition is in 9 hours. Your mentor walks in and says the team switched from XRP motors to TalonFX motors over the weekend. How many files do you have to change?

If your subsystem knew about the hardware directly: every single file that touched a motor. All of them. At 11pm. Good luck.

If your subsystem talks to an interface instead: one file. Swap the implementation. Go to sleep. Revolutionary.

Student
But can't I just use an if-statement? Like, if it's an XRP do this, else do that?
Also a valid question
You can! It will work. It will also mean your XRP code and your TalonFX code live in the same method, tangled together, each one quietly breaking the other in ways that only reveal themselves at competition. The IO pattern is just a cleaner if-statement that scales to a whole season without making you cry.
Student
Does every subsystem really need this? My indexer has one motor.
Yes. Especially that one.
One motor now. One motor, two sensors, a limit switch, and a current spike detector by week 4. The pattern costs you five minutes upfront and saves you two hours later. AdvantageKit also requires it for logging to work, so there's that.

The pattern in plain English

Three files. Three jobs. They never swap jobs.

DriveIO.java

The contract. Defines what a drivetrain can do. Knows nothing about actual hardware.

Think: job description

DriveIOXRP.java

The XRP version. Talks to real XRP hardware. Implements the contract.

Think: the person doing the job

DriveSubsystem.java

The boss. Calls methods on DriveIO. Has no idea who's on the other end. Doesn't care.

Think: the manager

What this looks like in code

public interface DriveIO {
    void updateInputs(DriveIOInputs inputs);
    void setVoltage(double left, double right);
    void stop();
}
This file has zero imports. It doesn't know about XRPDrivetrain. It doesn't know about TalonFX. It knows about nothing. That's the whole point. 
public class DriveIOXRP implements DriveIO {
    private final XRPDrivetrain drivetrain = new XRPDrivetrain();
    private final XRPGyro gyro = new XRPGyro();

    @Override
    public void setVoltage(double left, double right) {
        drivetrain.tankDrive(left / 12.0, right / 12.0);
    }
}
This one knows it's an XRP. It's the only one that gets to know that. Guard this privilege carefully. 
public class DriveSubsystem extends SubsystemBase {
    private final DriveIO io; // could be XRP, TalonFX, sim, anything

    public DriveSubsystem(DriveIO io) {
        this.io = io; // doesn't ask. doesn't care.
    }

    @Override
    public void periodic() {
        io.updateInputs(inputs);
        Logger.processInputs("Drive", inputs);
    }
}
Look at this class. Find the word XRP anywhere in it. You can't. This is the pattern working correctly.
There are no dumb questions

Q: What happens in simulation if I don't have a DriveIOSim?

You use an empty default implementation - a DriveIO that does nothing. The robot runs, nothing moves, and AdvantageKit still logs everything. Great for testing command logic without hardware.

Q: Do I write a new DriveIO interface for every robot?

No. One interface, multiple implementations. The interface is the contract - it doesn't change just because the hardware does. This is the whole point.

Q: What if I forget to implement a method from the interface?

Java will refuse to compile. Loudly. This is actually a feature - the compiler is your mean but honest friend who catches mistakes before the robot does.

Bronze - modify and observe

The code below drives both motors at 6 volts forward. Your job: change the numbers and watch what happens. This is science. Very low-stakes science. The robot is your lab. 

// In DriveSubsystem.java
public void driveForward() {
    io.setVoltage(6.0, 6.0); // left motor, right motor
}
  • Change both values to 3.0. Write down what you expect to happen before you run it. Then run it. Were you right?
  • Set left to 6.0 and right to -6.0. Describe what the robot does using only words your grandparent would understand.
  • Set both to 13.0. Does anything weird happen? Check the actual value the motor receives.
Voltage is how hard the motor pushes. 12V is full power. 0V is stopped. Negative reverses direction. Anything above 12V gets clamped - the motor has opinions about physics.
3V = slower (half speed). Left 6 / Right -6 = spin in place - this is called tank turning, not a malfunction. 13V gets clamped to 12V automatically - same as full speed. The hardware protects itself.
Silver - fill in the blanks

Someone deleted the most critical part of setVoltage(). The blanks are where the logic lived. The robot is currently very confused. 

@Override
public void setVoltage(double leftVolts, double rightVolts) {
    drivetrain.tankDrive(
        leftVolts  / _____,  // not a guess
        rightVolts / _____   // same number, promise
    );
}
  • What goes in the blanks? What are the units of tankDrive()? Check the docs before guessing.
  • Where did that number come from? Write one sentence explaining the reasoning.
  • Test it in sim. If the robot drives into the wall at full speed, something's wrong with your number.
tankDrive() wants values from -1.0 to 1.0 (percent output, not volts). A full battery is 12V. To convert volts to percent: divide by 12. This is the most important division you'll do today.
Both blanks = 12.0. tankDrive() wants -1.0 to 1.0. Dividing by 12 converts volts to that range. 6V / 12 = 0.5 = 50% speed. This math will come up again. Remember it.
Gold - build from scratch

Here's the contract. Your job is to fulfill it. No skeleton. No blanks. Just the interface, your knowledge, and Java's unforgiving compiler. 

// DriveIO.java - do not modify this file
public interface DriveIO {
    void updateInputs(DriveIOInputs inputs);
    void setVoltage(double left, double right);
    void stop(); // 0.0, not 1.0. learn from history.
}
  • Create DriveIOXRP.java implementing every method
  • Log velocity, position in meters, and gyro heading in updateInputs()
  • Verify in AdvantageScope - if velocity never changes, something is wrong with your logging, not the motor
  • Bonus: add a resetEncoders() method to the interface and both files. Justify why it belongs on the interface.
XRP encoders return inches - multiply by 0.0254 for meters. getAngleZ() gives gyro heading in degrees. If your robot drives sideways in sim, check which encoder method you're calling - there are two and they are not the same.

This code compiles perfectly. The robot boots without errors. And yet - something is deeply, specifically wrong with it. There are 3 bugs in the highlighted lines. Click each one to find out what past-you did and why future-you is not happy about it.

Bug 1 - The position is in the wrong units

AdvantageScope showed the robot traveled 47 meters. The gym is 28 meters long. Concerning.

getLeftDistanceInch() returns inches. Your input field is named leftPositionMeters. These are not the same unit. The robot logged a position wrong by a factor of 39.37, which is a fun number until you're trying to run an autonomous routine.

Fix
inputs.leftPositionMeters = drivetrain.getLeftDistanceInch() * 0.0254;

Bug 2 - Passing volts where the method wants percent

Commanded 6 volts. Robot received 6. tankDrive() interpreted this as 600% output and had a great time.

tankDrive() expects values from -1.0 to 1.0. Passing raw volts means passing values up to 12. The motor clamps to 1.0 (full speed) for anything above that. You've built a robot with two speeds: stopped and absolutely floored.

Fix
drivetrain.tankDrive(left / 12.0, right / 12.0);

Bug 3 - stop() works, but it works the wrong way

This one actually stops the robot. It just stops it in a way that will cause problems when you switch to real hardware.

Calling tankDrive(0, 0) works in XRP simulation, but it bypasses the hardware's native stop behavior. The idiomatic call that signals intent clearly - and lets the hardware handle braking correctly - is drivetrain.stopMotor(). Using tankDrive(0, 0) is like turning off your car by shifting to neutral and coasting instead of pressing the brake. Works, technically. Correct? Not quite.

Fix
drivetrain.stopMotor();
All 3 bugs found. Units mismatch, wrong value range, and a stop method with the wrong approach - these are real categories that FRC teams debug every season. You now recognize all three on sight. That's genuinely useful and it took you about 5 minutes. Good trade.
