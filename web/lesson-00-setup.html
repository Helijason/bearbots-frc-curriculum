Module 1 — Lesson 0

Setup + first drive

Get everything installed, run your first robot code, and watch a number move on a graph. The whole rest of the curriculum builds on this.
~2-3 hours
Hardware: your laptop only
Tools: WPILib, AdvantageScope, starter project

You're about to install two things, build a project you didn't write, and drive a robot that doesn't physically exist. By the end of today, all of that will work — and you'll never have to think about installation again for the rest of the curriculum.

Your brain is probably thinking this right now

"Two to three hours just to install some software? Really?"

Yes, really. The first time installing WPILib is slow. The first time setting up AdvantageScope is fiddly. The first time building a project, Gradle has to download dependencies. After today, all of this is fast — but today, it's slow. Plan for it.

What "done" looks like today

Six things need to work before you leave. We're going to do them in order. Each one builds on the last.

1. WPILib + VS Code installed

The official FRC editor with all the robot tools built in.

Includes Java, libraries, build tools — one big installer.

2. AdvantageScope installed

Separate app. Lets you see what the robot is doing in real time.

If WPILib is the workshop, AdvantageScope is the dashboard.

3. Starter project builds

Open the project, hit build, see BUILD SUCCESSFUL.

If this works, your environment is alive.

4. Simulator runs

Press a button, a window pops up, the robot is running.

No XRP needed today.

5. AdvantageScope shows values

Connect to the simulator, drag a value onto the graph, watch it move.

This is the moment everything clicks.

6. You can find the reference card

Two-page PDF cheat sheet. Saved somewhere you'll actually find it again.

You'll use this every session from now on.

Operating system support

Windows and Mac are supported. Instructions below cover both. The screenshots are Windows; Mac steps are called out where they differ.

Linux: see a mentor. We don't have time today to troubleshoot Linux-specific issues.

If your install fails

It happens. Java gets confused, firewalls block downloads, antivirus quarantines installers. If yours breaks and you've spent more than 15 minutes on one issue, switch to a backup laptop and keep going. You can troubleshoot your own machine later with a mentor.

The point of today is everyone leaves with a working setup — not necessarily on their own laptop.

What's on the USB drive

Everything you need is on the USB drive your mentor gave you. Copy all four files to your laptop before you start.

  • WPILib installer — Windows .exe or Mac .dmg
  • AdvantageScope installer — Windows .exe or Mac .dmg
  • Starter project ZIP — FRC-XRP-Starter.zip
  • Reference card — XRP-Reference-Card.pdf

Pick the right OS version. If you grab the wrong one, the installer just won't run.

Install WPILib + VS Code

WPILib is the official FRC software suite. It includes a special version of VS Code, the Java runtime you need, and every library used by FRC robot code. One installer, everything you need.

This is the slow one

The WPILib installer is about 1.5 GB and takes 10-15 minutes to extract and install. If nothing seems to be happening, it's probably still working. Don't cancel it.

On Windows

1. Double-click WPILib_Windows-2025.x.x.exe on your USB drive (the version number may differ)

2. If Windows asks "Do you want to allow this app to make changes?" → click Yes

3. If Windows SmartScreen blocks it → click More info, then Run anyway

4. Keep the default install path. Don't change it.

5. When asked: Download VS Code. Check every tool option (C++, Java, all of it).

6. Click Execute Install and wait. The progress bar moves slowly.

On Mac

1. Double-click WPILib_macOS-2025.x.x.dmg to mount it

2. Drag the WPILib installer to your Applications folder

3. Open it from Applications

4. If macOS says "cannot be opened because it is from an unidentified developer" → right-click the installer, choose Open, then click Open again

5. Same options as Windows: keep defaults, install all tools, download VS Code.

When the installer finishes, launch WPILib VS Code 2025 from your Start menu (Windows) or Applications folder (Mac). VS Code opens with a purple WPILib theme — that's how you know it's the right one.

Important: there are two VS Codes now

If you already had VS Code installed, you now have two of them. Regular VS Code and WPILib VS Code. They look almost identical, but only the WPILib one has the robot tools.

Always use WPILib VS Code 2025 for this curriculum. Pin it to your taskbar or dock so you don't accidentally open the wrong one.

Install AdvantageScope

AdvantageScope is a separate program from WPILib. It's how you'll see what the robot is doing — graphs of sensor values, the robot's position on a field, joystick state, console messages. It's the most-used tool in the curriculum after VS Code itself.

This is the easy one

AdvantageScope is a much smaller install — usually under 5 minutes. After the WPILib slog, this will feel quick.

On Windows

1. Double-click AdvantageScope-win-x64-3.x.x.exe

2. Follow the prompts — accept the defaults

3. When it finishes, AdvantageScope may launch automatically

On Mac

1. Double-click AdvantageScope-mac-3.x.x.dmg to mount it

2. Drag AdvantageScope to your Applications folder

3. Open it from Applications

4. If macOS blocks it: System Preferences → Security & Privacy → click Open Anyway

You should see the AdvantageScope main window: a workspace area in the middle, a timeline strip at the bottom, and a left sidebar (currently empty since nothing is connected yet).

There are no dumb questions

Q: Why do we need both VS Code and AdvantageScope?

VS Code is where you write code. AdvantageScope is where you see what the robot is doing. They're different tools for different jobs.

Q: When do we use the real XRP robot?

Lesson 3. Today is about getting your laptop ready. The simulator lets us write and test code before we touch any hardware.

Q: My install is stuck. Should I cancel?

If the progress bar has moved at all in the last 10 minutes, no — wait. If it's been frozen for more than 15 minutes with no disk activity, cancel and try again from the USB drive.

Brain power — before you move on

You're halfway done. Both tools are installed. Next we'll open the starter project and verify your environment can actually compile robot code.

The starter project is a complete, working robot program. You didn't write any of it. Today you're just opening it and verifying your environment can compile it. You'll explore what's actually inside in Lesson 1.

Step 1 — Extract the ZIP

  • Find FRC-XRP-Starter.zip on your USB drive
  • Copy it to your Documents folder (or wherever you keep code)
  • Right-click → Extract All on Windows, or double-click on Mac
  • You should now have a folder named FRC-XRP-Starter
Don't work from inside the ZIP

Some operating systems will let you "browse" inside a ZIP file like it's a folder. That's not the same as extracting. If you skip the extract step, nothing else in this lesson will work. Make sure you actually have a real folder, not just a window into the ZIP.

Step 2 — Open the folder in WPILib VS Code

  • Launch WPILib VS Code 2025 (the purple one — not regular VS Code)
  • Click File → Open Folder
  • Navigate to your extracted FRC-XRP-Starter folder
  • Click Select Folder

VS Code will ask if you trust the authors. Click Yes, I trust the authors.

You should see a file tree on the left with folders like src, files like build.gradle, and a .wpilib folder. If you see those, the project is open correctly.

Step 3 — Build the project

This is the moment of truth. If the build works, your environment is healthy.

How to build

1. Press Ctrl+Shift+P (Windows) or Cmd+Shift+P (Mac)

2. Type WPILib: Build Robot Code

3. Press Enter

4. Watch the terminal at the bottom of VS Code

The first build is slow — Gradle is downloading every library the project uses. Subsequent builds will be fast. When it finishes successfully, you'll see BUILD SUCCESSFUL in the terminal output.

What just happened

You compiled real robot code. The Java source files in src/main/java were turned into runnable bytecode. This is the same process you'll run every time you change anything in the project. You don't need to know how it works yet — just know that this is the loop: edit code, build, see if it works.

Build failed?

Check these in order:

1. Are you in WPILib VS Code (purple theme), not regular VS Code?

2. Did you open the folder, not just a single file?

3. Did the extract actually create a real folder?

4. Is your internet working? Gradle needs to download dependencies on first build.

If all of those are fine and it's still failing, ask a mentor. If you've burned 15 minutes on it, switch to a backup laptop.

This is the part you've been working toward. You're going to start the simulator, drive a virtual robot with your keyboard, and see logged values move in real time on a graph. After this, everything else in the curriculum is downhill.

Your brain is probably thinking this right now

"Why are we driving a robot that doesn't exist?"

Because 90% of bugs are logic bugs, not hardware bugs. The simulator catches them faster, cheaper, and without risk of the robot driving into the scoring table in front of the head referee. This is the same workflow real teams use to test code on competition robots.

Step 1 — Start the simulator

How to launch

1. In VS Code, press Ctrl+Shift+P (Windows) or Cmd+Shift+P (Mac)

2. Type WPILib: Simulate Robot Code

3. Press Enter

4. When prompted, check halsim_gui and click OK

5. Wait for the build (it's quick this time — already compiled)

6. A new window will appear: the Simulation GUI

Step 2 — Set up keyboard driving

Wire up the keyboard

1. Find the System Joysticks panel in the Simulation GUI

2. Drag Keyboard 0 from System Joysticks into the Joysticks panel (slot 0)

3. Find the Robot State panel → click Teleoperated → click Enable

4. Click on the Simulation GUI window so it has keyboard focus

W or ↑

Drive forward

S or ↓

Drive backward

A or ←

Turn left

D or →

Turn right

You won't see the robot move on screen

The Simulation GUI by itself doesn't show the robot moving — that's what AdvantageScope is for. The robot is receiving your commands. You'll see the proof in a moment.

Step 3 — Connect AdvantageScope

Keep the simulator running. Open AdvantageScope (don't close VS Code).

Connect to the simulator

1. In AdvantageScope: File → Connect to Simulator (or Ctrl+K)

2. Look at the bottom-left corner — it should say Connected

3. The left sidebar should populate with log keys (you'll see Drive/, AdvantageKit/, etc.)

Connect AdvantageScope BEFORE driving so you don't miss any data.

Step 4 — Drag values onto the graph

The left sidebar in AdvantageScope is now populated with every value your robot is logging. We'll graph two of them.

  • In the left sidebar, expand Drive
  • You'll see fields like LeftVelocityRadPerSec, RightVelocityRadPerSec
  • Drag LeftVelocityRadPerSec onto the main graph area
  • Drag RightVelocityRadPerSec onto the same graph

Step 5 — Drive and watch

  • Click back to the Simulation GUI window (so it has focus)
  • Make sure Teleoperated is still enabled
  • Press and hold W for a couple seconds, then release
  • Switch to AdvantageScope
  • The graph should show both velocity lines spike up while you held W, then fall back to zero
What you just did

You ran robot code without a robot. You drove a virtual machine with your keyboard. You logged sensor values to disk and graphed them in real time.

If you understand nothing else from today, understand this: code → build → simulate → drive → watch values is the loop. You'll run it hundreds of times.

Quick check

Why do we use the simulator instead of deploying to a real robot every time we test code?

Stuck? Common issues

Robot doesn't respond to keys: Click on the Simulation GUI window to give it focus. Make sure Teleoperated is still enabled (it disables when the simulator restarts).

AdvantageScope won't connect: The simulator must be running first. Check the bottom-left of AdvantageScope.

Values aren't changing: Did you drag the right fields? You want Drive/LeftVelocityRadPerSec and Drive/RightVelocityRadPerSec — not Setpoints.

Still stuck: Ask a mentor. This is a moment to get help, not to grind.

Two things left: save the reference card, then run through the checklist before you leave.

Save the reference card

The reference card is a two-page cheat sheet for the XRP curriculum. It has the most-used methods, the unit conversions you'll forget, and the core patterns for writing subsystems. You'll use it constantly. Memorizing it is not the goal — knowing where to look things up is.

  • Find XRP-Reference-Card.pdf on the USB drive
  • Copy it to the same folder where you extracted the starter project — that way it's right next to your code
  • Open it in your PDF reader to verify it works

Find these sections in the reference card

Spend two minutes locating each of the following. You don't have to read them today — just know they exist.

  • XRP Drivetrain methods — arcadeDrive(), getLeftEncoderDistance(), etc.
  • Common unit conversions — inches to meters, radians to degrees
  • AdvantageKit logging pattern — the three-line pattern in every subsystem
  • CommandScheduler call — the line that must be in robotPeriodic()
A habit to build

Before you ask a mentor "how do I do X?", check the reference card first. You'll get faster, and you'll start internalizing the patterns. The students who finish this curriculum strongest are the ones who build the habit of looking things up before asking.
