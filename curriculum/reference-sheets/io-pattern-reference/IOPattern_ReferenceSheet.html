Concept 1 — Java interfaces as contracts

BearBots FRC Curriculum  ·  Java | AdvantageKit | XRP  ·  Page 2 of 3

An interface defines what can be done. An implementation defines how. The subsystem only speaks to the interface — it never touches hardware directly.

Analogy — Subsystem, Interface, and Hardware Driver
Key insight DriveSubsystem never meets DriveIOXRP. Swap XRP for a swerve robot — the subsystem's method calls don't change. Zero edits to DriveSubsystem. Only the hardware driver gets replaced.
What this looks like in Java

Interface — the contract (no hardware, ever)

interface DriveIO {
  void setVoltage(double l, double r);
  void stop();
  void updateInputs(
      DriveIOInputsAutoLogged inputs);
}

Hardware Driver — one per hardware platform

class DriveIOXRP implements DriveIO {
  private final XRPDrivetrain dt =
      new XRPDrivetrain();

  public void stop() {
    dt.stopMotor(); // real hardware
  }
}

Concept 2 — Logging loop order & sanity check

BearBots FRC Curriculum  ·  Java | AdvantageKit | XRP  ·  Page 3 of 3

Three steps run inside periodic() in DriveSubsystem.java every 20ms. The order is not optional — swapping steps 1 and 2 silently corrupts replay.

What breaks if you swap steps 1 and 2 In REPLAY mode, AdvantageKit injects saved data at step 2. If step 1 runs after, it overwrites the injected data with zeros. Replay sees a robot that never moved — a bug that looks impossible to reproduce.
Why replay works REPLAY's DriveIO {} has no-op methods that write nothing to the struct. processInputs() feeds saved log values in instead. Same subsystem code. Different inputs source.
What the loop looks like in code — DriveSubsystem.java
// DriveSubsystem.java
@Override
public void periodic() {
    io.updateInputs(inputs);                 // step 1 — read hardware into struct
    Logger.processInputs("Drive", inputs);   // step 2 — log struct / REPLAY injects here
    double pos = inputs.leftPositionMeters;  // step 3 — safe to use inputs now
}
Quick sanity check
  • DriveSubsystem has zero hardware imports
  • updateInputs() before processInputs(). Always.
  • DriveIOXRP is the only file importing XRPDrivetrain
  • stop() sets motors to 0.0, not 1.0
  • Mode switch driven by Constants.currentMode
  • recordOutput() keys start with "Drive/…"
