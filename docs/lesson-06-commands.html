Read this once. Then start coding. Come back to it when you get stuck.

About the code in here

Most of it is made up. Method names like carryCommand or driveDistanceCommand are examples, not things sitting in your project waiting to be called. This reference teaches you the patterns. What your robot does, and what you name it, is your decision. Copying these examples line for line will not get you a working robot. Understanding them will.

1. What a command is

A subsystem is a piece of the robot: the Elevator, the Scoop, the Drive. A command is a job you give to a subsystem.

Every command answers four questions:

initialize()

What do I do when I start?

Runs once, right when the job begins.

execute()

What do I do while I run?

Runs 50 times a second while the job is going.

isFinished()

Am I done?

Returns true when the job is over.

end()

What do I do when I stop?

Runs once, when the job finishes or gets canceled.

Some commands finish on their own. "Drive to 0.3 meters" finishes when the elevator gets there. Some commands never finish on their own. "Keep reading the joystick" runs until something stops it.

Requirements

When you write a command, you say which subsystems it requires. This is the most important rule in the whole system:

The one rule you cannot break

Only one command can require a subsystem at a time.

If a command is using the Elevator and you start a new command that also wants the Elevator, the scheduler cancels the old one and runs the new one. It never runs both. That means two pieces of code can never fight over the same motor.

The thing that keeps track of all of this is called the CommandScheduler. It runs every 20 milliseconds and asks each running command: are you done yet? It handles the canceling for you. You never call it yourself.

2. Making commands: the four factories

You do not need to write a whole class every time you want a job. Your subsystem has four built-in factories. A factory is a method that builds a command for you. These four automatically require the subsystem they were called on.

// Do one thing, then finish immediately.
this.runOnce(() -> setAngle(90));

// Do a thing over and over, forever, until something cancels you.
this.run(() -> nudgeTargetHeight(0.001));

// Do one thing at the start, another thing at the end. Runs forever in between.
this.startEnd(() -> setVoltage(3), () -> setVoltage(0));

// Same as run, but also does something when you stop.
this.runEnd(() -> setVoltage(3), () -> setVoltage(0));

The () -> part is a lambda. It means "here is some code, but do not run it yet. Run it later when you need it." You are handing the command a set of instructions, not the answer.

Why factories instead of methods

You could write elevator.setHeight(0.3) as a plain method and call it. But then nothing owns the Elevator, and nothing stops two parts of your code from calling it at the same time. When you return a command instead, the scheduler knows who owns what and can cancel cleanly. That is the whole trade.

3. Putting commands together

Commands are Lego bricks. You snap them together and the thing you get back is also a command. That means you can snap that together with something else. It goes as deep as you want.

At the same time

elevator.goToHeightMetersCommand(0.3)
    .alongWith(scoop.setAngleCommand(90));
Both start now. The whole thing finishes when both are done.

One after the other

elevator.goToHeightMetersCommand(0.3)
    .andThen(scoop.setAngleCommand(0));
The elevator goes up. Only when it gets there does the scoop move.

With a pause in between

elevator.goToHeightMetersCommand(0.3)
    .andThen(Commands.waitSeconds(0.3))
    .andThen(scoop.setAngleCommand(0));
Elevator up, wait a third of a second for it to settle, then dump.

Race: first one done wins

Commands.race(driveForward(), waitForSensor());
Whichever finishes first ends the whole thing and cancels the other. Useful when you want "do this until that happens."
The rule that bites people

Once you put a command inside a group, it belongs to that group. You cannot also schedule it on its own somewhere else. If you try, the code crashes with an error. Build a fresh one instead. This is why factories are nice: calling scoop.setAngleCommand(90) twice gives you two separate commands.

4. Decorators

A decorator is a method you tack onto a command to change it. You get a new, modified command back.

// Give up after 2 seconds no matter what.
homeCommand().withTimeout(2);

// Stop as soon as this becomes true.
driveForward().until(() -> getDistance() > 1.0);

// Always run this when you stop, even if you got canceled.
goToHeightCommand(0.3).finallyDo(() -> holdCurrentHeight());

finallyDo is the safety net. If a command gets canceled halfway through, finallyDo still runs. Use it to make sure the robot ends in a safe state instead of whatever weird spot it got interrupted in.

5. Triggers: buttons to commands

A Trigger is a thing that is either true or false right now. A button being held is a trigger. So is "the elevator is above 0.2 meters."

You connect a trigger to a command with one of these:

BindingWhat happens
.onTrue(cmd) Fires once when the button is pressed. The command keeps going after you let go.
.whileTrue(cmd) Runs the command only while held. Cancels it the moment you release.
.onFalse(cmd) Fires once when you release.
.toggleOnTrue(cmd) Press to start. Press again to stop. 
controller.a().onTrue(scoop.setAngleCommand(45));
controller.rightBumper().onTrue(superstructure.carryCommand());
The single most common bug in this lesson

onTrue vs whileTrue is the difference between a light switch and a doorbell. Pick the wrong one and the robot does the wrong thing even though the code compiles fine.

onTrue starts a job and walks away. whileTrue holds the job for as long as you hold the button. If you want a jog button that stops when you release, use whileTrue. If you want a button that sets a position and stays there, use onTrue.

Combining triggers

A trigger is just a true/false question, so you can mix them like you mix questions in real life.

// Only score if the bumper is held AND the elevator is high enough.
controller.rightBumper().and(elevator.isAboveScoringHeight())
    .onTrue(superstructure.scoreCommand());

// Ignore accidental taps. Button must be held 0.2 seconds to count.
controller.a().debounce(0.2).onTrue(scoop.setAngleCommand(45));
These don't exist yet

isAboveScoringHeight() does not exist. Neither does scoreCommand(). They are examples of things you might decide your robot needs, and then write. That is true of most method names in this section. The pattern is what you are learning. The names are yours to pick.

.and(), .or(), and .negate() build new triggers out of old ones. You can also build a trigger out of anything at all:

new Trigger(() -> elevator.getHeightMeters() > 0.25)
    .onTrue(Commands.print("Elevator is high!"));

6. Default commands

There is one more way a command can start. A subsystem can have a default command: the job it does whenever nothing else is asking for it.

drive.setDefaultCommand(
    drive.run(() -> drive.arcadeDrive(
        -controller.getLeftY(), -controller.getRightX())));

That is how driving works. No button, no trigger. The Drive subsystem is idle, so the scheduler starts its default command, which reads the sticks forever.

The magic part

If you schedule an auto-drive command, it requires the Drive, so the default gets canceled. When the auto-drive finishes, the Drive is idle again, so the scheduler restarts the default automatically. Stick control comes right back. You wrote zero lines of code to switch between them.

Rules for default commands:

  • It must require that subsystem and nothing else.
  • It should never finish on its own. If it does, the scheduler just restarts it, which is usually not what you meant.
  • Not every subsystem needs one. The Elevator holds its setpoint in periodic() already, and the Scoop is a servo that holds its angle by itself. Neither needs a default.

7. The jog pattern

You want to hold a button and have the elevator creep up. Let go and have it stay put.

controller.povUp().whileTrue(
    elevator.run(() -> elevator.nudgeTargetHeight(kJogRateMetersPerLoop)));

controller.povDown().whileTrue(
    elevator.run(() -> elevator.nudgeTargetHeight(-kJogRateMetersPerLoop)));

Read it out loud: while the up button is held, run this over and over: bump the target a little higher.

Here is the part worth slowing down for. You are not moving the elevator. You are moving the elevator's target. The PID controller in periodic() is always chasing the target, whether or not any command is running. So:

  • Hold the button, target climbs, PID chases it, elevator rises.
  • Let go, command ends, target stops moving, PID parks the elevator right there and holds it against gravity.

If you had sent voltage directly instead, letting go would mean zero volts, and the elevator would sag down. Moving the target gets you a free hold. That is why we do it this way.

Why "per loop," not "per second"

kJogRateMetersPerLoop is per loop, not per second. The robot loops 50 times a second, so a rate of 0.002 meters per loop is 0.1 meters per second. The name says "per loop" so nobody accidentally multiplies by time twice.

Clamp inside the subsystem

public void nudgeTargetHeight(double deltaMeters) {
  targetHeightMeters = MathUtil.clamp(
      targetHeightMeters + deltaMeters, kMinHeight, kMaxHeight);
}
This is real. nudgeTargetHeight exists on Elevator.java.

The clamp lives in the subsystem, not in the button binding. Hold the button at the top and the target just pins against the max. It cannot run away past the top of the elevator and leave the real elevator behind, which would mean a long ugly wait when you finally jog back down.

8. The Superstructure

The Elevator knows about the elevator. The Scoop knows about the scoop. Neither of them knows about the other, and it should stay that way.

But "carry a ball" is not an elevator job or a scoop job. It is a robot job that uses both. That job needs a home, and that home is the Superstructure.

The Superstructure holds both subsystems and hands out named actions:

public class Superstructure {
  private final Elevator elevator;
  private final Scoop scoop;

  public Superstructure(Elevator elevator, Scoop scoop) {
    this.elevator = elevator;
    this.scoop = scoop;
  }

  public Command carryCommand() {
    return elevator.goToHeightMetersCommand(kCarryHeight)
        .alongWith(scoop.setAngleCommand(kCarryAngle));
  }

  public Command scoreCommand() {
    return elevator.goToHeightMetersCommand(kScoreHeight)
        .andThen(Commands.waitSeconds(0.3))
        .andThen(scoop.setAngleCommand(kDumpAngle));
  }

  public Command stowCommand() {
    return elevator.goToHeightMetersCommand(kStowHeight)
        .alongWith(scoop.setAngleCommand(kFlatAngle));
  }
}
Illustrative — this doesn't exist yet

carryCommand, scoreCommand, and stowCommand are made-up names. Superstructure.java does not exist in your project. This shows the pattern. The actual commands your robot needs, and what you call them, are your decisions.

That is the whole idea. The Superstructure does not touch motors. It does not have an IO layer. It just knows the recipes: which subsystem commands to combine, in what order, with what timing.

Notice carryCommand uses alongWith and scoreCommand uses andThen. That is a real engineering decision, not a style choice. Carry can happen all at once because nothing collides. Scoring has to wait for the elevator to settle before dumping, or the ball goes to the wrong place.

How this plays with your jog buttons

This is the payoff. Watch:

  1. You are jogging the elevator up with the D-pad.
  2. You press the bumper for carryCommand.
  3. carryCommand requires the Elevator. The scheduler sees the jog command also requires the Elevator, so it cancels the jog.
  4. carryCommand drives to the carry setpoint and finishes.
  5. You press the D-pad again. The jog command starts fresh, nudging from wherever the elevator is now.
Why we built all of this

You did not write a single line of code to make that work. No mode flag, no isManualMode boolean, no if-statement asking whether it is okay to move. The requirement system did all of it. This is why we bothered with subsystems and requirements in the first place. Everything you have built for five lessons was setting up this moment.

9. Autonomous is just a command

Autonomous is not a special mode with special rules. It is one command that gets scheduled when the match starts.

Because compositions are commands, your whole auto routine is one expression:

public Command scoreAndParkAuto() {
  return drive.driveDistanceCommand(0.5)
      .andThen(superstructure.scoreCommand())
      .andThen(drive.driveDistanceCommand(-0.3));
}
Nothing in that block exists yet

driveDistanceCommand, scoreCommand, scoreAndParkAuto: those are all methods someone decided to write. Your Drive subsystem does not have a drive-a-distance command until you add one. Look at what Orbit Odyssey actually scores, decide what your robot needs to do, then build the pieces and snap them together. The shape above is the shape yours will have. The parts are your call.

Every routine you write is a Command, so you can put several in a chooser and pick one from the dashboard before the match:

autoChooser.addDefaultOption("Score and Park", scoreAndParkAuto());
autoChooser.addOption("Just Park", drive.driveDistanceCommand(0.5));

Give every auto step a timeout. If a mechanism jams, .withTimeout(2) moves you on instead of freezing the rest of the routine for the whole match.

superstructure.scoreCommand().withTimeout(2)

10. Seeing what your commands are doing

You cannot fix what you cannot see. AdvantageScope shows you which commands are running, right now, in a list. When the robot does something weird, look there first. Nine times out of ten you will see a command you thought was finished still sitting there, or a command that got canceled the instant it started because something else grabbed the subsystem.

Give your commands names so that list is readable:

superstructure.carryCommand().withName("Carry");

Without a name you get something like ParallelCommandGroup, which tells you nothing when there are four of them.

11. Mistakes everyone makes

Using onTrue when you meant whileTrue. Press the jog button, elevator runs to the ceiling, never stops. onTrue starts a job and walks away.

Reusing a command object. Save a command in a variable, put it in two groups, and the code crashes. Call the factory each time instead. scoop.setAngleCommand(90) builds a fresh one every call.

A default command that finishes. It restarts instantly, forever, and initialize() fires 50 times a second. If your subsystem twitches, check this.

Forgetting requirements. If you use Commands.run() (the static one) instead of this.run() (your subsystem's), you have to pass the subsystem in yourself or nothing owns it and two commands will fight.

Doing math in the button binding. Clamps, limits, and unit conversions belong inside the subsystem. The binding should say what you want, not how to do it safely.

12. Where this goes on a real FRC robot (preview only)

Some robots have parts that will smash into each other if you move them in the wrong order. On those robots, the Superstructure gets bigger. It holds an enum of every legal state (STOW, INTAKE, SCORE_HIGH), and a periodic() method that only allows moves between states that are safe. Big teams like 6328 and 254 build them this way.

The XRP does not have that problem. The elevator and scoop cannot hit each other. So we use command composition, which does the same job with a fraction of the code.

When to reach for the state machine

Reach for the state machine when the robot forces you to. Not before.
