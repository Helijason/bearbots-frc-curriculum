Every subsystem you've built so far - Arm, Scoop - moves the same way: you pick a setpoint, and code either jumps straight there or slews toward it at a fixed rate per program loop. That is what a hobby servo does. The controls for angle setpoints are built into the servo. There is no flexibility in the control.

Your brain is probably thinking this right now

"Can't I just do the same thing - set a target height, ramp toward it?"

Try it and the carriage either crawls up too slowly to be useful, or slams into the top of its travel and stalls the motor against the hard stop. A servo has its own internal controller doing this work for you. A bare DC motor on a spool does not - there's nothing between "voltage in" and "however fast the motor decides to spin" except gravity, friction, and whatever your code chooses to send. You need to close the loop yourself giving the subsystem a lot of flexibility.

Open-loop vs. closed-loop

The motor is made up of two main components. first the DC motor and an encoder. When using an open-loop setter it tells the motor "spin at this voltage" and never checks whether that produced the right result. A closed-loop controller measures the actual position, compares it to the target, and continuously corrects the error. That correction loop - measure, compare, correct, repeat every 20ms - is what "closed-loop control" means.

Servo (Lesson 04 pattern)

You command an angle. The servo's own internal electronics hold it there. Your code never sees an error term.

DC motor + encoder (today)

You command a voltage. Position is whatever physics does with that voltage. Your code has to read the encoder and correct itself, every loop, forever.

Why gravity is different this time

An arm swinging on a servo doesn't fight a spool-and-cable elevator's constant downward pull. Holding still costs voltage - even at the target, zero volts means falling.

PIDController - "Error Correction" half

WPILib's PIDController takes a measurement and a setpoint, and returns an output correction:

kP - proportional

Correction scales with how far off you are right now. Bigger error, bigger push. Too high and the carriage overshoots and oscillates; too low and it never quite gets there.

kI - integral

Correction accumulates from error that's persisted over time. Fixes steady-state droop. Almost always start at zero - it's the gain most likely to make things worse if you don't need it.

kD - derivative

Correction reacts to how fast the error is changing. Damps oscillation from a too-high kP. Also usually starts at zero.

Feed forward - "Expected voltage" half

PID alone reacts to error after it happens. Feedforward adds the voltage you already know you'll need, before there's any error to react to - so PID only has to correct the leftover mistake, not do all the work.

The three feedforward terms

kS - static friction. The minimum voltage to overcome stiction and get the mechanism moving at all.

kG - gravity. The voltage needed just to hold position with zero velocity. For an elevator this is never zero - gravity never stops pulling.

kV - velocity. Extra voltage proportional to how fast you want to move.

The snippet you'll use calls feedforward.calculate(0.0) - it asks for zero velocity feedforward because the goal is holding a height, not tracking a moving trajectory. That still leaves kS and kG doing real work every single loop.

Student
If kV is for velocity and we always pass 0.0, why does it exist at all?
Fair question
Because ElevatorFeedforward is a general-purpose class, not one built only for holding still. If you later drive the elevator with a motion profile (a target velocity that isn't zero), kV matters immediately. For today, holding a fixed height, it's along for the ride - tune it last, after kS and kG are solid.

Live tuning - and why you delete the tune code when you're done

Guessing four to six gains cold is miserable. The aksubsystem-motor snippet wires each gain to a LoggedNetworkNumber under /Tuning/Elevator/, so you can updaate gains in AdvantageScope, graph apropriate values while the robot runs and watch the response change in real time.

Tune handles are scaffolding, not the final product

The snippet's periodic() re-reads all six LoggedNetworkNumbers and rebuilds the controller and feedforward every single loop - that's deliberately wasteful so you can retune live without redeploying. Once you land on gains you like, copy the values into ElevatorConstants, then coment out or delete the LoggedNetworkNumber fields and the two lines in periodic() that read them. Competition code doesn't rebuild a controller 50 times a second for no reason.

See AdvantageScope's Tuning Mode documentation for exactly how to toggle tuning mode and edit values live once connected to your simulator or robot.

Homing - why an encoder alone isn't enough

An motor's encoder counts relative motion from wherever it was powered on. It has no idea where "zero" actually is unless you tell it. homeCommand() solves this by driving gently toward the bottom hard stop, watching for the motor to stall (moving, then suddenly not), and calling that position zero.

Reading the stall detector

The BooleanSupplier inside homeCommand() tracks a moved flag so it can't falsely trigger before the carriage starts moving (velocity starts at exactly 0, which would otherwise look identical to "stalled"). Only after velocity has exceeded kHomingMovingMetersPerSec does it start checking for a drop back below kHomingStallMetersPerSec.

// Copyright (c) 2026 BearBots FRC Team 6964
// Open Source Software; you can modify and/or share it under the terms of
// the MIT License available in the root directory of this project.

package frc.robot.subsystems.elevator;

import static frc.robot.Constants.kLoopPeriodSecs;
import static frc.robot.Constants.kMaxBatteryVoltage;
import edu.wpi.first.math.MathUtil;

public class ElevatorIOSim implements ElevatorIO {
  private double appliedVolts = 0.0;
  private double positionMeters = 0.0;
  private double velocityMetersPerSec = 0.0;

  @Override
  public void updateInputs(ElevatorIOInputs inputs) {
    // Gravity pulls the carriage down: it takes voltage just to hold position.
    double effectiveVolts = appliedVolts - ElevatorConstants.kGravityVolts;

    // Linear model: velocity scales with effective voltage.
    velocityMetersPerSec =
        (effectiveVolts / kMaxBatteryVoltage)
            * ElevatorConstants.kSimMaxSpeedMetersPerSec;

    // Integrate velocity into position each loop tick.
    positionMeters +=
        velocityMetersPerSec * kLoopPeriodSecs;

    // Stop at the physical limits; carriage can't sink through the bottom or fly off the top.
    double clamped =
        MathUtil.clamp(
            positionMeters,
            ElevatorConstants.kMinHeightMeters,
            ElevatorConstants.kMaxHeightMeters);
    if (clamped != positionMeters) {
      velocityMetersPerSec = 0.0;
    }
    positionMeters = clamped;

    inputs.appliedVolts = appliedVolts;
    inputs.positionMeters = positionMeters;
    inputs.velocityMetersPerSec = velocityMetersPerSec;
    inputs.encoderCount =
        (int) (positionMeters / ElevatorConstants.kEncoderDistancePerPulseMeters);
  }

  @Override
  public void setVoltage(double volts) {
    appliedVolts = volts;
  }

  @Override
  public void resetEncoder() {
    positionMeters = 0.0;
    velocityMetersPerSec = 0.0;
  }

  @Override
  public void stop() {
    appliedVolts = 0.0;
  }
}
The sim model integrates velocity into position by hand, every loop - this is the physics a real motor+gearbox+encoder gives you for free. Note it clamps to the constants' min/max height and zeroes velocity at the hard stops, mimicking a real carriage that can't pass through its own frame.

// Copyright (c) 2026 BearBots FRC Team 6964
// Open Source Software; you can modify and/or share it under the terms of
// the MIT License available in the root directory of this project.

package frc.robot.subsystems.elevator;

import edu.wpi.first.math.MathUtil;
import edu.wpi.first.math.controller.ElevatorFeedforward;
import edu.wpi.first.math.controller.PIDController;
import edu.wpi.first.wpilibj2.command.Command;
import edu.wpi.first.wpilibj2.command.SubsystemBase;
import java.util.function.BooleanSupplier;
import org.littletonrobotics.junction.AutoLogOutput;
import org.littletonrobotics.junction.Logger;
import org.littletonrobotics.junction.networktables.LoggedNetworkNumber;
import static frc.robot.Constants.kMaxBatteryVoltage;

import frc.robot.Constants;

public class Elevator extends SubsystemBase {
  private final ElevatorIO io;
  private final ElevatorIOInputsAutoLogged inputs = new ElevatorIOInputsAutoLogged();

  // One controller does the math: error in, volts out. Tolerance decides "close enough".
  private final PIDController controller = new PIDController(
      Constants.currentMode == Constants.Mode.SIM
          ? ElevatorConstants.kPSim
          : ElevatorConstants.kPReal,
      Constants.currentMode == Constants.Mode.SIM
          ? ElevatorConstants.kISim
          : ElevatorConstants.kIReal,
      Constants.currentMode == Constants.Mode.SIM
          ? ElevatorConstants.kDSim
          : ElevatorConstants.kDReal);

  // Rebuilt each loop from the tune handles below (ElevatorFeedforward gains are final).
  private ElevatorFeedforward feedforward = new ElevatorFeedforward(
      Constants.currentMode == Constants.Mode.SIM
          ? ElevatorConstants.kSVoltsSim
          : ElevatorConstants.kSVoltsReal,
      Constants.currentMode == Constants.Mode.SIM
          ? ElevatorConstants.kGVoltsSim
          : ElevatorConstants.kGVoltsReal,
      Constants.currentMode == Constants.Mode.SIM
          ? ElevatorConstants.kVVoltSecPerMeterSim
          : ElevatorConstants.kVVoltSecPerMeterReal);

  // TUNE handles. Bake winners into constants, then delete these + the setPID/feedforward lines.
  private final LoggedNetworkNumber kP =
      new LoggedNetworkNumber("/Tuning/Elevator/kP", controller.getP());
  private final LoggedNetworkNumber kI =
      new LoggedNetworkNumber("/Tuning/Elevator/kI", controller.getI());
  private final LoggedNetworkNumber kD =
      new LoggedNetworkNumber("/Tuning/Elevator/kD", controller.getD());
  private final LoggedNetworkNumber kS =
      new LoggedNetworkNumber("/Tuning/Elevator/kS", feedforward.getKs());
  private final LoggedNetworkNumber kG =
      new LoggedNetworkNumber("/Tuning/Elevator/kG", feedforward.getKg());
  private final LoggedNetworkNumber kV =
      new LoggedNetworkNumber("/Tuning/Elevator/kV", feedforward.getKv());

  private double targetMeters = ElevatorConstants.kDefaultHeightMeters;
  private boolean homing = false;

  public Elevator(ElevatorIO io) {
    this.io = io;
    controller.setTolerance(ElevatorConstants.kHeightToleranceMeters);
  }

  // Single source of truth. Every setpoint, named or raw, lands here.
  public void setHeightMeters(double meters) {
    targetMeters = MathUtil.clamp(
        meters,
        ElevatorConstants.kMinHeightMeters,
        ElevatorConstants.kMaxHeightMeters);
  }

  // Jump to a height immediately. Bind to a button:
  //   button.onTrue(elevator.setHeightMetersCommand(ElevatorConstants.kTopHeightMeters));
  public Command setHeightMetersCommand(double meters) {
    return runOnce(() -> setHeightMeters(meters));
  }

  // Go to a height and finish once within tolerance. Use this inside sequences.
  public Command goToHeightMetersCommand(double meters) {
    return run(() -> setHeightMeters(meters))
        .until(this::atGoal)
        .finallyDo(interrupted -> {
          // Cancelled early: hold where we are instead of chasing the old target.
          if (interrupted) setHeightMeters(inputs.positionMeters);
        });
  }

  // Stall-stop homing. Drives toward the hard stop, zeros the encoder there.
  public Command homeCommand() {
    // Stall-stop, but only after the carriage has actually started moving,
    // otherwise the initial velocity of 0 ends it on the first loop.
    BooleanSupplier stalled = new BooleanSupplier() {
      boolean moved = false;
      public boolean getAsBoolean() {
        double v = Math.abs(inputs.velocityMetersPerSec);
        if (v > ElevatorConstants.kHomingMovingMetersPerSec) moved = true;
        return moved && v < ElevatorConstants.kHomingStallMetersPerSec;
      }
    };
    return startRun(
            () -> homing = true,
            () -> io.setVoltage(ElevatorConstants.kHomingVolts))
        .until(stalled)
        .withTimeout(ElevatorConstants.kHomingTimeoutSecs)
        .finallyDo(() -> {
              io.stop();
              io.resetEncoder();
              setHeightMeters(ElevatorConstants.kDefaultHeightMeters);
              homing = false;
            });
  }

  @AutoLogOutput(key = "Elevator/AtGoal")
  public boolean atGoal() {
    return controller.atSetpoint();
  }

  public void stop() {
    io.stop();
  }

  @Override
  public void periodic() {
    io.updateInputs(inputs);
    Logger.processInputs("Elevator", inputs);

    if (homing) return;   // homing owns the motor; skip PID this loop

    Logger.recordOutput("Elevator/TargetMeters", targetMeters);

    // TUNE: apply live gains each loop. Delete when you bake.
    controller.setPID(kP.get(), kI.get(), kD.get());
    feedforward = new ElevatorFeedforward(kS.get(), kG.get(), kV.get());

    // Feedback (PID) + feedforward (gravity/static hold). velocity setpoint 0 = hold.
    double volts = controller.calculate(inputs.positionMeters, targetMeters)
        + feedforward.calculate(0.0);
    volts = MathUtil.clamp(volts, -kMaxBatteryVoltage, kMaxBatteryVoltage);

    io.setVoltage(volts);
  }
}
This is the whole control loop in one file: PID for feedback, feedforward for the voltage you already know you need, six LoggedNetworkNumber tune handles, and a homing command that takes over the motor when active. No visualization code here yet - Part 3 adds it.

    new JoystickButton(controller, XboxController.Button.kRightBumper.value)
        .onTrue(elevator.setHeightMetersCommand(ElevatorConstants.kTopHeightMeters));
    new JoystickButton(controller, XboxController.Button.kLeftBumper.value)

        .onTrue(elevator.homeCommand());
  }
Right bumper sends the carriage to top, left bumper to bottom, Start runs the homing routine. There's no getter for scoop or elevator yet - both come back in Part 3, once there's a pose to expose.

Compose the poses in Robot.java
▶

This is the one place in the whole project that's allowed to know about both Elevator and Scoop at once. Three things happen in robotPeriodic(): feed the elevator's height into the scoop's 2D root, and compose the scoop's local 3D pose onto the elevator's pose using Transform3d instead of just standing it at the origin.

Robot.java — import
import edu.wpi.first.math.geometry.Pose3d;
import edu.wpi.first.math.geometry.Transform3d;
Robot.java — robotPeriodic()
    CommandScheduler.getInstance().run();

    // Scoop mounts to the elevator carriage — shift its Mechanism2d root to match, so the
    // 2D scoop ligament visually rises and lowers with the elevator.
    robotContainer.getScoop().setBaseHeightMeters(robotContainer.getElevator().getPositionMeters());

    // Aggregate all component poses for AdvantageScope 3D view.
    Logger.recordOutput("FinalComponentPoses", new Pose3d[] {
    // Elevator mounts to the frame; scoop mounts to the elevator carriage, so its pose
    // rides on top of the elevator's (composition happens here, not inside either subsystem).
    Pose3d elevatorPose = robotContainer.getElevator().getComponentPose();
    Pose3d scoopLocalPose = robotContainer.getScoop().getComponentPose();
    Pose3d scoopPose = elevatorPose.transformBy(
        new Transform3d(scoopLocalPose.getTranslation(), scoopLocalPose.getRotation()));

    Logger.recordOutput("FinalComponentPoses", new Pose3d[] {
      robotContainer.getArm().getComponentPose(),
      elevatorPose,
      scoopPose
    });
  }
Transform3d is how WPILib composes one pose relative to another. elevatorPose.transformBy(...) takes the scoop's own local pose (computed inside Scoop.java with no idea the elevator exists) and re-expresses it in field-relative terms, on top of wherever the elevator carriage currently is. This is the same composition pattern you'll use constantly once robots have more than two or three moving mechanisms stacked on each other.

Build and simulate. Open AdvantageScope's Mechanism tab and drag on Elevator/Mechanism2d and Scoop/Mechanism2d - the scoop ligament should visibly rise and fall as the elevator moves. Then open a 3D Field tab and add FinalComponentPoses - all three components (arm, elevator, scoop) should track correctly, with the scoop riding the elevator carriage.

Three tiers, one elevator. Bronze tunes by feel. Silver adds a new named setpoint. Gold tunes feedforward and can explain what each term is doing. Pick where you are; Gold subsumes the others.

Bronze - tune kP by feel

Connect AdvantageScope's Tuning Mode to your simulator. Find /Tuning/Elevator/kP and drag its slider while commanding the elevator between top and bottom with the bumpers.

  • Drop kP to something clearly too low (e.g. 5.0). Command a height change. Describe what you see - does it get there? How fast?
  • Raise kP well above the default (e.g. 150.0). Command a height change again. What's different? Listen for the motor - does it sound different?
  • Find a kP value where the carriage reaches its target quickly without visibly overshooting or oscillating. Write down the value.
  • With kP at your chosen value, home the elevator (Start button) and confirm it still settles at the bottom cleanly.
Watch the carriage, not just the number. Oscillation looks like the carriage overshooting the target and bouncing back and forth around it before settling - that's kP too high. Never quite arriving, or arriving very slowly, is kP too low.
Silver - add a new named height setpoint

Three setpoints exist: bottom, middle, top. Add a fourth and wire a button to it.

  • Open ElevatorConstants.java. Add a new named setpoint following the existing pattern, e.g. public static final double kScoringHeightMeters = 0.100;
  • Open RobotContainer.java. Bind a currently-unused button (check what's already bound before picking one) to elevator.setHeightMetersCommand(ElevatorConstants.kScoringHeightMeters)
  • Build, simulate, press your new button. Confirm the carriage moves to the expected height and the AdvantageScope Elevator/TargetMeters plot shows the new value.
Check kMaxHeightMeters before picking a value - anything above it gets silently clamped by setHeightMeters, which will look like your new button does nothing.

Gold - tune feedforward and explain kG

PID alone can hold a height, but watch what it costs. Isolate feedforward's contribution, then tune it properly.

  • With the elevator at rest at a mid-height, set /Tuning/Elevator/kG to 0.0 in AdvantageScope's Tuning Mode. Does the carriage sag or drift? What is PID doing to compensate?
  • Slowly raise kG from zero while watching the carriage hold a fixed height. Find the value where the carriage holds steady with minimal correction effort - watch Elevator/TargetMeters vs. the actual measured position for how hard PID is working.
  • Once kG is dialed in, try raising kS slightly. Command very small height changes near the current position - does the carriage respond more crisply to tiny commands?
  • Bake your final kP, kG, and kS values into ElevatorConstants.java, then delete the six LoggedNetworkNumber tune-handle fields and the two lines in periodic() that read them.
  • Explain out loud: what would happen if you deployed to the real robot with kG still at 0.0? Why does an elevator need this term when Lesson 04's Arm and Scoop never did?
kG is the voltage that exactly cancels gravity at rest. If it's too low, PID's kP term has to constantly fight gravity just to hold still, which shows up as either a small steady-state droop (needs kI to fully fix) or visible strain if kP is pushed high to compensate. If it's too high, the carriage will drift upward at rest instead of holding position. The correct value holds a mid-height with the smallest possible PID correction.
