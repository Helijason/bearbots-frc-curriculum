In Lesson 03 you built Scoop.java the simple way: one file, XRPServo imported directly, servo called straight from periodic(). It works. Run it today and the scoop moves.

The problem shows up when you try to test at home with no XRP, or when you need to replay a match log to find a bug. The simple version can't do either. The IO pattern can.

Your brain is probably thinking this right now

"The scoop already works. Why are we changing it?"

Because "works when plugged in" and "works at competition when something breaks at 9pm" are two different things. The IO pattern is how serious FRC teams get to the second one.

Four files, four jobs

Every subsystem using the IO pattern has at minimum four files. You already saw this in the Arm subsystem from Lesson 03:

ScoopIO.java

The contract. Defines what a scoop can do. Has no idea what hardware exists.

Think: job posting

ScoopIOInputs

A struct of sensor values the IO layer reads each loop cycle. Annotated with @AutoLog so AdvantageKit logs everything automatically.

Think: the form the hardware fills out

ScoopIOXRP.java

The XRP version. Talks to the real servo. Implements the contract.

Think: person doing the job on XRP

ScoopIOSim.java

The simulation version. Tracks commanded state in memory. No hardware needed.

Think: person doing the job in sim

Then Scoop.java itself: the subsystem. It only knows about ScoopIO, never about the servo. Same structure you saw in Arm.java in Lesson 03.

Read the Arm files first

Read the Arm files before making any new files. The Scoop files are nearly identical. Click any annotated line to open the explanation.

ArmIO.java - the interface
The contract. Every method here is a capability the arm hardware must support, regardless of which hardware that is.
1 @AutoLog ⬤
@AutoLog is an AdvantageKit annotation. At compile time it generates a sibling class called ArmIOInputsAutoLogged that adds toLog() and fromLog() methods automatically. You never write those methods and never see the generated file. Without this annotation, none of your inputs get logged.
2 public static class ArmIOInputs { ⬤
A plain Java class with public fields. The IO implementation writes to these fields inside updateInputs() every loop cycle. Because of the @AutoLog annotation above, every field here is logged automatically. Add a field and it shows up in AdvantageScope on the next build with no extra logging code.
3   public double commandedAngleDeg = 0.0; -
6 public interface ArmIO { ⬤
A Java interface defines method signatures without implementations. Any class that says implements ArmIO must provide a body for every method listed here — if it doesn't, the compiler stops immediately. The interface has no hardware imports at all: no XRPServo, no TalonFX, nothing.
7   default void updateInputs(ArmIOInputs inputs) {} ⬤
updateInputs() is called by the subsystem every loop cycle to pull fresh sensor data. The default keyword means implementing classes can skip overriding it — the default body is empty. That's what makes the replay no-op stub work: pass in new ArmIO() {} and updateInputs() does nothing, because replay mode reads from the log directly.
8   default void setAngle(double angleDeg) {} ⬤
Every action the hardware supports gets a method here. setAngle() takes an angle in degrees. The interface says nothing about how that angle gets applied - whether it's divided by a servo range, clamped, or converted to encoder ticks. That decision lives entirely in the implementation. The interface only cares about what, never how.
ArmIOXRP.java - the XRP implementation
The only file that knows what hardware the arm uses. Everything hardware-specific lives here and only here.
1 public class ArmIOXRP implements ArmIO { ⬤
implements ArmIO is the promise that this class will provide a body for every method in the interface. Leave out setAngle() and the compiler stops you. No runtime surprises.
2   private final XRPServo servo = new XRPServo(4); ⬤
XRPServo is declared here and only here. Search every other file in the subsystem folder: ArmIO.java, ArmIOSim.java, Arm.java. None of them import XRPServo. This is the pattern working correctly. If you ever see XRPServo imported in Arm.java, the pattern is broken.
5–10   @Override public void setAngle(double angleDeg) { // ... ⬤
@Override tells Java this method is fulfilling a contract from the interface. Notice there is no clamping here. The IO layer just assigns the value and calls the servo. Clamping happens in Arm.java, at the subsystem boundary, before the angle reaches this method. kMinAngleDeg and kMaxAngleDeg from ArmConstants.java are used there, not here.
ArmIOSim.java - the simulation implementation
No hardware. Tracks state in memory so the subsystem can run and log in simulation.
3   private double commandedAngleDeg = ArmConstants.kStowedAngleDeg; ⬤
In sim there's no servo to read back from, so the implementation just remembers the last angle it was told to set. updateInputs() copies that value into the inputs struct so it gets logged. AdvantageScope shows the commanded angle changing when you press buttons, even with no hardware connected.
6–8   @Override public void updateInputs(ArmIOInputs inputs) { // ... ⬤
inputs.commandedAngleDeg = commandedAngleDeg; — that's the entire body. It copies the remembered value into the struct. The subsystem just calls updateInputs() and reads whatever comes back; it has no way to distinguish this from a real servo.

REPLACE - Scoop.java (aksubsystem-servo)
// Copyright (c) 2026 BearBots FRC Team 6964
// Open Source Software; you can modify and/or share it under the terms of
// the MIT License available in the root directory of this project.

package frc.robot.subsystems.scoop;

import edu.wpi.first.math.MathUtil;
import edu.wpi.first.wpilibj2.command.Command;
import edu.wpi.first.wpilibj2.command.SubsystemBase;
import org.littletonrobotics.junction.Logger;
import frc.robot.Constants;

public class Scoop extends SubsystemBase {
  private final ScoopIO io;
  private final ScoopIOInputsAutoLogged inputs = new ScoopIOInputsAutoLogged();

  private double targetAngleDeg = ScoopConstants.kDefaultAngleDeg;

  public Scoop(ScoopIO io) {
    this.io = io;
  }

  // Single source of truth. Every setpoint, named or raw, lands here.
  public void setAngleDeg(double angleDeg) {
    targetAngleDeg = MathUtil.clamp(
        angleDeg,
        ScoopConstants.kMinAngleDeg,
        ScoopConstants.kMaxAngleDeg);
  }

  // Jump to a setpoint immediately. Bind to a button:
  //   button.onTrue(xxx.setAngleDegCommand(XxxConstants.kDeployedAngleDeg));
  public Command setAngleDegCommand(double angleDeg) {
    return runOnce(() -> setAngleDeg(angleDeg));
  }

  // Slew to a setpoint at a capped rate, finish within tolerance.
  public Command setAngleDegSteppedCommand(double angleDeg, double degPerSec) {
    double target = MathUtil.clamp(
        angleDeg,
        ScoopConstants.kMinAngleDeg,
        ScoopConstants.kMaxAngleDeg);
    return run(() -> {
      double step = degPerSec * Constants.kLoopPeriodSecs;
      double delta = MathUtil.clamp(target - targetAngleDeg, -step, step);
      setAngleDeg(targetAngleDeg + delta);
    }).until(() -> Math.abs(target - targetAngleDeg)
        < ScoopConstants.kAngleToleranceDeg);
  }

  @Override
  public void periodic() {
    io.updateInputs(inputs);
    Logger.processInputs("Scoop", inputs);
    Logger.recordOutput("Scoop/TargetAngleDeg", targetAngleDeg);

    io.setAngleDeg(targetAngleDeg);
  }
}
Search this file for XRPServo. Not there. The subsystem talks to ScoopIO — it doesn't know or care what's on the other end. The clamp lives here, at the subsystem boundary, not in the IO layer.

The scoop moves in the real world, but AdvantageScope can also show a live 2D diagram that updates as you press buttons. This is the Mechanism2d view. It lets you confirm your angle mapping is correct without looking at the robot.

This part makes changes to three files:

  • Scoop.java — add a LoggedMechanism2d and a getComponentPose() getter
  • RobotContainer.java — add a getScoop() getter
  • Robot.java — aggregate both subsystem poses into one FinalComponentPoses key
Why does FinalComponentPoses move to Robot.java?

AdvantageScope's 3D view reads a single array under the key FinalComponentPoses. If each subsystem logs its own array to that key, the last one written each loop wins and the others disappear. The fix is to collect all component poses once per loop and log them together. Robot.java's robotPeriodic() runs after all subsystem periodic() methods, making it the right place for that aggregation.

Why the 2D view is the one to trust right now

The scoop's 3D chassis offset is still a placeholder of (0.0, 0.0, 0.0) until it is measured on the physical robot, so in the 3D Field view the scoop sits at the chassis origin instead of its real mounting point. The rotation is correct, the position is not. The 2D Mechanism2d view does not depend on that offset, so it is the trustworthy confirmation that your angle mapping is right.

If nothing moves

You have a log file from Part 5. Replay mode lets you re-run your robot code against that exact log, getting the same sensor inputs byte for byte, without touching any hardware. This is how you debug a match at home at midnight.

Why replay matters

Normal debugging: something breaks at competition, you try to reproduce it at practice, it doesn't happen, and you're guessing.

With replay: load the log from the match where it broke, run your code against it, and watch it happen again on your laptop with no robot and no field. The order of those two lines in periodic() is what makes that possible.

How replay works

In REPLAY mode, processInputs() doesn't write to a log. It reads from one, overwriting the inputs struct with values from the original run. Your IO implementation (ScoopIOXRP, ScoopIOSim) is replaced with a no-op stub. The hardware doesn't run. The log does.

Normal run (REAL or SIM)
io.updateInputs(inputs);
// reads from hardware or sim

Logger.processInputs("Scoop", inputs);
// writes inputs to log file
Replay run (REPLAY)
io.updateInputs(inputs);
// no-op: does nothing

Logger.processInputs("Scoop", inputs);
// reads from log INTO inputs struct

Your subsystem code runs normally. It reads from inputs. It computes outputs. It calls Logger.recordOutput(). Those outputs go into a new log file as ReplayOutputs/. You then compare RealOutputs/ from the original run against ReplayOutputs/ from the replay.

Switching modes

Two knobs control what the robot code does, and you set both by hand. Constants.currentMode picks the runtime behavior (which IO runs, whether the logger reads or writes). The ROBOT_MODE line in build.gradle picks which simulation extensions Gradle launches. They use slightly different names: the hardware mode is REAL in Constants but XRP in build.gradle.

What you are doing Constants.currentMode build.gradle ROBOT_MODE
Simulator on your laptop Mode.SIM RobotMode.SIM
Live XRP hardware (bonus) Mode.REAL RobotMode.XRP
Replaying a log Mode.REPLAY RobotMode.REPLAY

Whenever you finish, set both back to the SIM row. That is the safe state to commit and the state your next simulate session expects.

Run your first replay

Follow the instructor first

Watch the full walkthrough on the instructor's screen before doing it yourself. The sequence matters.

1
Set the runtime mode. In Constants.java, change currentMode to Mode.REPLAY. 
public static final Mode currentMode = Mode.REPLAY;  // ← changed
2
Set the build mode. In build.gradle, change ROBOT_MODE to RobotMode.REPLAY.

3
List your logs. In the terminal, run ./replay.bat -List. It prints the log files in logs/.
4
Run the replay. Copy the filename from the list and run ./replay.bat filename with just the name, no path. The script blanks any leftover extension settings, sets the log path, runs your code against the log, and writes a new _sim.wpilog next to the original.
5
Compare in AdvantageScope. Open the new _sim.wpilog. Find ReplayOutputs/Scoop/CommandedAngleDeg and compare it to AdvantageKit/RealOutputs/Scoop/CommandedAngleDeg from the original run. They should match exactly: same values, same timestamps.
6
Add a new output and replay the same log. This is the part that makes replay useful. In Scoop.java, add a new logged value in periodic() right after line 68: 
68    Logger.recordOutput("Scoop/Goal", goal.toString());
69    Logger.recordOutput("Scoop/NewAngle", clamped + 5.0);
Save the file. The watch loop from step 4 is still running: it detects the change and replays the log again on its own, so the new value shows up in AdvantageScope without you running anything. You did not touch the robot or the field. You computed a brand new output from the inputs that were recorded the first time.
7
Find the new output. There is now a ReplayOutputs/Scoop/NewAngle key that did not exist in the original log. Compare it to ReplayOutputs/Scoop/CommandedAngleDeg: it tracks the same shape, offset by exactly 5 degrees. The original run never logged this value, and you produced it anyway, from recorded data, with no robot in the room.
8
Stop the replay. The replay runs in a watch loop, so it keeps going until you stop it. In the terminal, press Ctrl+C. When it asks Terminate batch job (Y/N)?, type Y and press Enter.
9
Set both knobs back to SIM when you are done: Constants.currentMode = Mode.SIM and ROBOT_MODE = RobotMode.SIM. You can keep the NewAngle line or remove it; it does not change how the robot drives.
Gotchas worth knowing

In Part 4 you checked Sim GUI in the extension picker. Replay is the opposite: it must load no extensions at all. AdvantageKit refuses to replay if any simulation extension is loaded, which is exactly why replay runs through ./replay.bat instead of the picker. Use the script, not Simulate Robot Code.

The _sim.wpilog file is the replay output, written next to the source log. replay.bat -List hides *_sim.wpilog on purpose, so you cannot accidentally replay a replay.

Commit with both knobs on the SIM row. A committed REPLAY or XRP setting breaks the next person’s simulate session.

If they don't match

If RealOutputs/ and ReplayOutputs/ don't match with unchanged code, replay isn't reading the log correctly. Confirm both knobs are on the REPLAY row and that you launched with ./replay.bat rather than Simulate Robot Code.

Do I have to change Constants.java back after replay?
Yes. Every time.
Leaving it on REPLAY and deploying means the robot runs with no hardware outputs, no motor commands. Set both knobs back to the SIM row before your next simulate session, and to the REAL/XRP row before deploying to hardware.

What if I didn't log enough? Like, what if the thing that broke isn't in the log?
Then you're out of luck for that match.
That's why logging discipline matters. If the value isn't in the inputs struct, replay can't reproduce it. If processInputs() wasn't called, the log is empty. Any time you add a sensor to a subsystem, add it to the inputs struct in the same commit.
