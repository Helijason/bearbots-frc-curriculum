You just ran WPILib: Create a new project and VS Code generated a bunch of files. Your instinct is probably to start editing them immediately. Resist that instinct for about 20 minutes. Let's understand what we have first.

Your brain is probably thinking this right now

"I just want to make the robot drive. Why are we reading files?"

Fair. But if you don't know what these files do, you'll spend the rest of the season randomly editing things hoping something works. Understanding the structure takes 20 minutes. Debugging blind takes hours. Your call.

Creating a new project the right way

Always use the WPILib tools — never create robot files by hand. The tools set up the package declarations, imports, and class structure correctly every time.

How to create a new WPILib project

1. Press Ctrl+Shift+P to open the command palette

2. Type WPILib: Create a new project and press Enter

3. Select: Template → Java → Command Robot Skeleton (Advanced)

4. Enter your team number, project name, and save location

5. Click Generate Project

VS Code will open the new project automatically. Wait for the build to finish before touching anything.

The files — what each one does

A freshly generated WPILib project has exactly these files. Click each one to learn its job.

Main.java  click to expand

The entry point. Starts the robot.

Job: The very first thing that runs when the robot boots. It calls RobotBase.startRobot(Robot::new) which hands control to your Robot class.

Should you modify it? Almost never. The only reason to touch this file is if you rename your main Robot class.

The rule

Do not add static variables or initialization code to Main.java. The comment at the top says this. It means it.

Robot.java  click to expand

The lifecycle manager. Controls what happens in each robot mode.

Job: Contains the lifecycle methods — robotPeriodic(), autonomousInit(), teleopInit(), etc. These are called automatically by WPILib at the right times.

Extends: In a standard project, Robot extends TimedRobot. In our AdvantageKit project, it extends LoggedRobot instead. LoggedRobot extends TimedRobot — so all the same lifecycle methods work.

The most important line: CommandScheduler.getInstance().run() in robotPeriodic(). This is what makes the entire command-based framework work. Without it, no commands ever execute.

@Override
public void robotPeriodic() {
    CommandScheduler.getInstance().run(); // don't remove this
}

RobotContainer.java  click to expand

The switchboard. Creates subsystems and wires up buttons.

Job: This is where you instantiate subsystems, set up controller bindings, and provide the autonomous command. It's the "main switchboard" of the robot.

Why not put this in Robot.java? Separation of concerns. Robot.java handles timing and lifecycle. RobotContainer handles what the robot actually does. Keeping them separate makes both easier to read and debug.

This is the file you'll edit most. Every new subsystem and every button binding lives here.

Constants.java  click to expand

The number vault. Motor IDs, speeds, PID values — all in one place.

Job: Stores constants that are used throughout the robot code. Motor IDs, gear ratios, PID gains, field dimensions. Anything that's a fixed number that might need to change should be here — not hardcoded in a subsystem.

In AdvantageKit projects, Constants.java also holds the Mode enum that controls REAL/SIM/REPLAY behavior.

Why does this matter?

Imagine your motor ID is hardcoded as 4 in five different files. Someone changes the wiring. Now you have to find and change it in five places without missing any. With Constants.java you change it once. Sleep well.

build.gradle  click to expand

The build recipe. Tells Java what libraries to include.

Job: Configures the build system. Specifies which WPILib version to use, which vendor libraries to include, and how to compile and deploy the code.

When you add AdvantageKit using WPILib: Manage Vendor Libraries, it adds entries to this file automatically. You rarely edit it by hand.

If your build fails mysteriously, build.gradle is often the culprit — mismatched versions or missing vendor deps.

How the files connect

The files aren't independent — they call each other in a specific order every loop cycle.

Student
So Robot.java creates RobotContainer, and RobotContainer creates subsystems?
Exactly right
Robot.java creates one RobotContainer in its constructor. RobotContainer creates all the subsystems. Then every 20ms, Robot.java calls robotPeriodic(), which runs the CommandScheduler, which calls periodic() on every subsystem. That's the whole loop.
Student
What's LoggedRobot vs TimedRobot? The template gives me TimedRobot.
Good catch
LoggedRobot extends TimedRobot — it adds AdvantageKit's logging wrapper around each loop cycle. When we add AdvantageKit to our project, we change extends TimedRobot to extends LoggedRobot. All the lifecycle methods work the same way. We just get logging for free.
There are no dumb questions

Q: Can I put everything in Robot.java and skip the other files?

Technically yes. It will work for about two weeks. Then you'll have 800 lines of Robot.java, two programmers editing the same file, and merge conflicts that make everyone cry. The structure exists for a reason.

Q: What's the vendordeps folder?

JSON files that tell the build system where to download vendor libraries like AdvantageKit, CTRE Phoenix, or REV hardware. When you run WPILib: Manage Vendor Libraries, it creates these files automatically.

Q: Do I need to understand build.gradle?

Not deeply. Know that it exists, know that vendor libraries affect it, and know that if the build breaks mysteriously — check here first.

Before you ever touch the XRP, you need to be comfortable in the simulator. This is where you'll spend most of your development time. Real hardware is for verifying — simulation is for building.

Your brain is probably thinking this right now

"The simulator isn't a real robot. Why would I care about simulated data?"

Because 90% of bugs are logic bugs, not hardware bugs. The simulator catches them faster, cheaper, and without risk of the robot driving into the scoring table in front of the head referee.

Step 1 — Launch the simulator

How to launch

1. Press Ctrl+Shift+P

2. Type WPILib: Simulate Robot Code and press Enter

3. A dialog appears asking which extensions to load. Check halsim_gui and click OK

4. Wait for the build. The Robot Simulation window opens automatically.

Common mistake

If the simulation window opens but nothing seems to work — check the System Console panel first. Errors print there. "Nothing is happening" and "there is an error" look identical from the outside.

Step 2 — Understand the simulation GUI panels

Robot State

Enable/disable the robot. Switch between Teleop, Auto, and Test modes.

If you forget to enable — nothing moves. This is the #1 sim confusion.

Joysticks

Map keyboard or real controller inputs to robot joystick slots.

Keyboard 0 is always available. Drag it to Joystick[0].

System Console

Shows print statements, errors, and stack traces from your robot code.

Always check here first when something goes wrong.

NetworkTables

Shows all values being published to NetworkTables in real time.

AdvantageKit data appears here as "AdvantageKit/..."

Step 3 — Connect AdvantageScope

Connect AdvantageScope to the simulator

1. Open AdvantageScope (install from GitHub if not already installed)

2. File → Connect to Simulator (or press Ctrl+K)

3. The left panel populates with log keys — you should see AdvantageKit/ appear

4. Connect AdvantageScope BEFORE enabling the robot — otherwise you miss the first seconds of data

Step 4 — Set up keyboard driving

You can drive the simulated robot using your keyboard. Here's the exact setup:

Keyboard joystick setup

1. In the Simulation GUI, find the Joysticks panel

2. Find Keyboard 0 in the available inputs list

3. Drag it to the Joystick[0] slot

4. Go to Robot State panel → select Teleoperated → click Enable

5. Click back on the Simulation GUI window (keyboard input only works when this window is focused)

Keyboard default mappings

W - Forward (Left stick Y negative)

S - Backward (Left stick Y positive)

A - Turn left (Left stick X negative)

D - Turn right (Left stick X positive)

These match the default arcade drive command in RobotContainer.

Using a real controller instead

1. Plug in your USB controller BEFORE launching the simulator

2. It appears automatically in the Joysticks panel

3. Drag it to Joystick[0]

4. Left stick Y = forward/backward, Left stick X = turn

Move the sticks and watch the axis values change in the Joysticks panel to confirm it's working.

Step 5 — Watch data in AdvantageScope

Viewing live data while driving

1. In AdvantageScope's left panel, find the Drive/ folder and expand it

2. Drag LeftPositionMeters to the graph area on the right

3. Drag GyroYawDegrees to a second graph tab

4. Drive with keyboard or controller

5. Watch the graphs update in real time as the robot moves

If graphs are flat while driving - updateInputs() has a bug.

If graphs jump immediately to big numbers - check unit conversion (inches vs meters).

Do this right now

Launch the simulator. Connect AdvantageScope. Enable the robot in Teleop mode. Drive using the keyboard. Watch at least one value move in AdvantageScope.

Don't move on until you've done this. The rest of the curriculum assumes you're comfortable with this workflow.

Bug 1 - extends TimedRobot instead of LoggedRobot

The comment says "logging will work fine here." The comment is lying.

TimedRobot doesn't know anything about AdvantageKit. Extending it means Logger.start() will fail silently or throw an error, and none of your inputs will ever be logged. The fix is one word: change TimedRobot to LoggedRobot. LoggedRobot extends TimedRobot — everything else works the same.

Fix
public class Robot extends LoggedRobot {

Bug 2 - CommandScheduler never runs

The TODO has been there since the project was created. It will be there forever if no one fixes it.

robotPeriodic() runs every 20ms but does nothing. Without CommandScheduler.getInstance().run(), no commands ever execute — not autonomous, not teleop driving, not button bindings. The robot boots, logs nothing, and sits there judging you.

Fix
CommandScheduler.getInstance().run();

Bug 3 - RobotContainer recreated on every teleop start

"Fresh start!" it says cheerfully, destroying all your subsystems and button bindings every time teleop begins.

Creating new RobotContainer() in teleopInit() throws away the subsystems and bindings created in the constructor and builds new ones. Any state carried from autonomous is lost. Any commands running are orphaned. RobotContainer should be created exactly once — in the Robot constructor. Remove this line entirely.

// Remove line 17 entirely. RobotContainer is created once in the constructor.
