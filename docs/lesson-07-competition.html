Last lesson, your DriveDistance command set the motors to 0.5 and waited for the encoder to read 1 meter. It worked - sort of. The robot probably overshot to 1.05m, or stopped early at 0.94m. Now we fix that with feedback.

Today is also the lesson where every concept from Modules 1 and 2 lands: subsystems, the IO pattern, logging, commands, sensors. By the end you'll be able to read real team code and recognize what's going on.

Your brain is probably thinking this right now

"My auto worked at 0.5 power. Why complicate it?"

It worked once, on a fresh battery, on a clean carpet, going forward. Drop the battery to 11.5V - now 0.5 power moves slower. Run the same code on a slightly rougher surface - you stop short. Run it backward instead of forward - the robot moves at a different speed because the gearbox has different friction. Open loop is "set it and pray." Closed loop measures and adjusts.

Open loop vs closed loop - the punchline

Open loop - "set and pray"

drive.drive(0.5, 0.0)

You command 50% power. The motor does whatever 50% means today - depends on battery, friction, weight, surface, direction.

No correction. If you're off, you stay off.

Fine for: simple movements, prototyping, when accuracy doesn't matter.

Closed loop - "measure and adjust"

error = target - current; setOutput(error * kP);

You command a goal. The code reads sensors, compares to the goal, and adjusts the motor output every cycle (50 times per second).

Self-correcting. If it's off, self-correcting.

Required for: anything that has to be precise. Tuning determines how well it corrects.

The three sensors on the XRP - what each tells you

Encoders

"How far has each wheel turned?"

Two of them, one per side. Counts pulses as the wheel rotates - converts to inches via a gear-ratio constant.

Use for: distance traveled, wheel speed, odometry.

XRP gives getLeftDistanceInch() and getRightDistanceInch().

Gyro

"Which way is the robot pointing?"

Single axis on the XRP - reports heading in degrees relative to where it was when last reset.

Use for: turning to a heading, holding a heading while driving, detecting collisions.

XRP gives gyro.getAngle() for heading.

Reflectance

"Is there a line under the sensor?"

Two analog sensors on the bottom - high reading on dark surfaces, low on light. Used for line following.

Use for: detecting tape lines, edges of game elements, the auto starting line.

XRP gives left and right reflectance values 0.0–1.0.

Reading sensors - through the IO layer, always

You learned this in Lesson 04 but it bears repeating: your subsystem doesn't read sensors directly. It reads them through the IO layer, which fills an inputs struct, which gets logged. Today we add three more fields to that struct.

@AutoLog
public class DriveIOInputs {
    // ... existing fields from earlier lessons ...
    public double leftPositionMeters = 0.0;
    public double rightPositionMeters = 0.0;

    // new for Lesson 07:
    public double leftVelocityMetersPerSec = 0.0;
    public double rightVelocityMetersPerSec = 0.0;
    public double headingDegrees = 0.0;
}

Then in DriveIOXRP.updateInputs(), fill them from the hardware:

inputs.leftPositionMeters     = drivetrain.getLeftDistanceInch() * 0.0254;
inputs.rightPositionMeters    = drivetrain.getRightDistanceInch() * 0.0254;
inputs.leftVelocityMetersPerSec  = drivetrain.getLeftEncoderRate() * 0.0254;
inputs.rightVelocityMetersPerSec = drivetrain.getRightEncoderRate() * 0.0254;
inputs.headingDegrees         = gyro.getAngle();
Units - this is where bugs live

The XRP reports distances in inches. Your code probably wants meters. 1 inch = 0.0254 meters. If you forget the conversion, your "1 meter" auto will travel 39 meters - off the field, through the wall, into the parking lot. The reference card has the conversions; use them.

Why log velocities AND positions?

Position tells you "how far you've gone." Velocity tells you "how fast you're going." When debugging, you'll want both. If position looks right but velocity is jumpy, you have a sensor noise problem. If velocity looks right but position is wrong, you've got a sign or units bug. Log both, debug faster.

You have sensors. You have a target. The bridge between them is a controller - code that runs every cycle, reads the current sensor value, compares to the target, and decides what motor output to set. The simplest useful controller is proportional control, also known as P control. Once you understand P, you understand 80% of the controllers FRC teams use.

Your brain is probably thinking this right now

"PID. I've heard about PID. Three letters, three terms, super-complicated."

P is the only term you need today. The "I" and "D" are extensions for special cases - useful, but optional. Real teams ship competition code with just P all the time. Master P first; the others add complexity for specific problems.

The P in PID - one line of math

Proportional control says: the further you are from the target, the harder you push. Halfway there? Push half as hard. At the target? Don't push at all.

double error = target - current;        // how far off are we?
double output = error * kP;             // push proportional to error
drive.setVoltage(output);                // apply it

That's it. Three lines. kP is a constant you tune - "how aggressively do I want to correct?" Start with 1.0 and adjust based on what the robot does.

Tuning kP - what the symptoms tell you

kP too small

Robot creeps to the target slowly. Or never quite reaches it - "stuck at 0.92 meters when target is 1.0." The push isn't strong enough to overcome friction.

Fix: increase kP. Try doubling it.

kP just right

Robot drives to the target smoothly, slows down as it approaches, settles within a few millimeters in under a second.

You found it. Stop tuning.

kP too big

Robot slams toward the target, overshoots, comes back, overshoots the other way. Oscillates. In extreme cases never settles.

Fix: cut kP in half.

The tuning loop

1. Start with kP = 1.0

2. Run the auto. Watch in AdvantageScope: graph target alongside current.

3. Too slow? Double kP. Too oscillatory? Halve it.

4. Repeat until current converges smoothly to target.

This is the actual workflow on real robots. AdvantageScope is what makes it possible - you can see the response curve and react.

A complete closed-loop DriveDistance

Here's DriveDistance from Lesson 06, rewritten with proportional control. Same four lifecycle methods, just smarter execute().

public class DriveDistance extends Command {
    private final DriveSubsystem drive;
    private final double targetMeters;
    private double startMeters;

    private static final double kP        = 4.0;        // volts per meter of error
    private static final double tolerance = 0.02;       // 2 cm

    public DriveDistance(DriveSubsystem drive, double meters) {
        this.drive = drive;
        this.targetMeters = meters;
        addRequirements(drive);
    }

    @Override public void initialize() {
        startMeters = drive.getLeftPositionMeters();
    }

    @Override public void execute() {
        double traveled = drive.getLeftPositionMeters() - startMeters;
        double error    = targetMeters - traveled;
        double volts    = error * kP;
        drive.setVoltage(volts, volts);          // drive both sides equal
    }

    @Override public boolean isFinished() {
        double traveled = drive.getLeftPositionMeters() - startMeters;
        return Math.abs(targetMeters - traveled) < tolerance;
    }

    @Override public void end(boolean interrupted) {
        drive.stop();
    }
}
What's different from Lesson 06?

execute() - instead of setting motors to a fixed 0.5, computes an error and pushes proportionally.

isFinished() - instead of "have we passed the target?", asks "are we within tolerance of the target?" The robot might stop slightly before or after; tolerance covers both.

Everything else - the four-method structure, addRequirements, end() stopping the motors - is identical to Lesson 06. The framework didn't change. Just the brain.

WPILib's PIDController - same math, less code

Once you understand P, you'll find WPILib already wrote it for you. PIDController is a class that does the math, plus tolerance checking and reset handling. Use it when your code starts getting more than 5 lines of error-and-output math.

private final PIDController pid = new PIDController(4.0, 0.0, 0.0);  // kP, kI, kD
// ...
@Override public void execute() {
    double output = pid.calculate(drive.getLeftPositionMeters(), targetMeters);
    drive.setVoltage(output, output);
}
@Override public boolean isFinished() {
    return pid.atSetpoint();                // uses configured tolerance
}

Same behavior as the hand-rolled version, but you can pass kI and kD later if you need them. Most teams use this from day one. Now you know what it's doing under the hood.

The other two letters - in 60 seconds

I - Integral

"What if I keep persistently being a little off?" Sums the error over time and adds it to the output. Solves "I always settle 5cm short" by accumulating that error and pushing harder.

Use when you have steady-state error you can't tune away with P alone.

D - Derivative

"How fast is my error changing?" Subtracts a term proportional to the rate of change. Damps oscillation - the robot brakes itself as it approaches the target.

Use when P alone is oscillating and you can't reduce kP without going too slow.

You will not need I or D today. Most teams don't need them this season. If P is working, ship it.

Bronze - replace open-loop with proportional control

Take your existing DriveDistance command and convert it from "set 0.5 power" to "compute error and push proportionally."

  • Open DriveDistance.java
  • Add a kP constant. Start with 4.0.
  • Replace execute() with the three-line P-control math
  • Update isFinished() to use a tolerance check (Math.abs(error) < 0.02)
  • Run a 1-meter auto. Open AdvantageScope. Graph Drive/LeftPositionMeters against your target.
  • Tune kP until the robot settles at the target without overshooting
If your robot doesn't move at all, your kP might be too small - the error times kP isn't producing enough voltage to overcome static friction. Try kP = 8.0. If it overshoots and oscillates, halve kP. The reference implementation in Part 2 uses kP = 4.0; your XRP might need different.
Silver - close the gyro loop while driving forward

Your DriveDistance probably drifts - one motor is slightly stronger than the other, and the robot curves left or right. Add a second proportional controller using the gyro to correct heading drift.

  • In initialize(), also record the starting heading from the gyro
  • In execute(), compute heading error: currentHeading - startHeading
  • Compute a rotation correction: headingError * kHeading (try kHeading = 0.05)
  • Set left motor to baseVoltage - rotationCorrection, right to baseVoltage + rotationCorrection
  • Run on the carpet. Verify the robot drives straighter than before.
The signs matter. If the robot drifts left, you need to push the left side and cut the right to correct. If your correction makes the drift worse, flip the signs. Don't fight the math - flip the variable. 
private double startHeading;
private static final double kP = 4.0;
private static final double kHeading = 0.05;

@Override public void initialize() {
    startMeters = drive.getLeftPositionMeters();
    startHeading = drive.getHeadingDegrees();
}

@Override public void execute() {
    double traveled       = drive.getLeftPositionMeters() - startMeters;
    double distanceError  = targetMeters - traveled;
    double baseVolts      = distanceError * kP;

    double headingError   = drive.getHeadingDegrees() - startHeading;
    double rotation       = headingError * kHeading;

    drive.setVoltage(baseVolts - rotation, baseVolts + rotation);
}
Gold - closed-loop TurnToAngle from scratch

Use WPILib's PIDController to build a TurnToAngle that turns to a precise heading and stops. This is the version you'd actually use in competition.

  • Create TurnToAngle.java, extends Command, requires drive
  • Constructor takes target degrees
  • Create a PIDController(kP, 0.0, 0.0) as a field. Start with kP = 0.05.
  • Call pid.setTolerance(2.0) in the constructor (degrees)
  • initialize(): record start heading; call pid.reset()
  • execute(): double rot = pid.calculate(currentHeading - startHeading, targetDegrees); set left/right motors to -rot, +rot
  • isFinished(): return pid.atSetpoint();
  • end(): drive.stop();
  • Add to AutoChooser. Verify it turns within 2° of target.
If pid.atSetpoint() never returns true, the robot is oscillating around the target without settling - it briefly satisfies the tolerance but bounces out. setTolerance takes a position tolerance and an optional velocity tolerance; passing both stops atSetpoint() from triggering until the robot is within tolerance AND not moving fast. Try pid.setTolerance(2.0, 5.0).

Before you touch the field, prove it works in sim. Before you commit your code, record what you tuned and why. These two tools are your gate and your memory.

Sim verification checklist — complete before every field run

The rule

If it doesn't pass in sim, it doesn't run on the field. No exceptions. Catching a bug in sim costs 30 seconds. Catching it on the field costs your next match slot.

Sim verification checklist

1. Build succeeds — Ctrl+Shift+P → WPILib: Build Robot Code → BUILD SUCCESSFUL

2. Simulator launches — no errors in System Console on startup

3. Auto command initializes — in AdvantageScope, Commands folder shows your auto command starting

4. Auto command ends — Commands folder shows the command finishing cleanly, not timing out

5. Position curve looks right — graph Drive/LeftPositionMeters — smooth approach, settles near target, no oscillation

6. Motors stop — after command ends, velocity drops to zero in AdvantageScope

All six pass? You're cleared for the field.

What the curve tells you

Before you run anything on the field, predict what the AdvantageScope position curve will look like. Sketch it in your head: rises smoothly, approaches the target, slows near it, settles. Then run it. If the curve matches your prediction — your mental model is right. If it doesn't — that mismatch is exactly where the bug is. Prediction → run → compare. That's the tuning loop.

Field tuning log — one row per run

Every field run gets a row. If you don't write it down, you won't remember whether kP = 3.0 or 4.0 was the one that worked — and you'll retune the same thing twice.

What to record each run

kP: what value did you use?

PARK_DISTANCE_METERS: what was the constant?

Result: where did it stop? (in zone / short / long / crashed)

Curve shape: smooth / oscillating / creeping / slammed past

Next change: what one thing are you changing and why?

The one-change rule

Change exactly one constant per run. If you change kP and distance at the same time and the result improves, you don't know which one fixed it. You don't know which one to keep.

Science: change one variable, observe the result, conclude. Same process.

What "3 consecutive successful runs" means

Don't declare victory after one good run. Run it three times from the same starting position. If it lands in the zone all three times — you're done tuning, write down the values, commit the code. If it misses on run 2 or 3 — you're not done. One lucky run isn't reliability.
